# SAP 拡張 設計判断記録 (v2)

本ドキュメントは SAP 拡張パックの設計判断（ADR）を記録する。各判断の「なぜ」を残すことで、将来の変更時に根拠を参照できるようにする。

対象読者: テンプレートメンテナー、新規参画者。

関連ドキュメント:
- [SAP_GLOSSARY.md](SAP_GLOSSARY.md) -- 用語の定義
- [SAP_CONFIG_REFERENCE.md](SAP_CONFIG_REFERENCE.md) -- 設定ファイルリファレンス
- [MANIFEST.yaml](../MANIFEST.yaml) -- 拡張パック全体構成

## ADR-001: AC の SSoT を basic_design.md に一本化

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2025-12 |
| **カテゴリ** | データモデル / トレーサビリティ |

**背景:**
初期設計では AC が basic_design.md、spec.md、plan.md に分散定義されており、各ファイル間で AC の粒度や表現が乖離する問題が頻発していた。

**決定:**
basic_design.md の `traceability_rows` を AC の唯一の正本（SSoT）とし、spec.md は basic_design.md から AC を参照・展開する形に統一した。v2 では Step 1-A1（AI 駆動の yaml → basic_design.md 転記）が AC-ID、TS-ID、Task-ID を構造化データとして転記する。

**理由:**
- AC が 1 箇所にしかないため、変更時の追跡が容易
- 構造化された traceability_rows により ID の重複・欠番を防止
- spec.md の AC は basic_design.md の「詳細化」であり、追加・削除はしない

**トレードオフ:**
- basic_design.md が肥大化しやすい（traceability_rows が長くなる）
- basic_design.md の承認後に AC を追加する場合、Gate 1,2 の再承認が必要

## ADR-002: Phase 1.5（SAP コンテキスト取得）の挿入

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2025-11 |
| **カテゴリ** | ワークフロー |

**背景:**
SAP 開発では、テーブル構造・権限オブジェクト・既存ソースコードなど、SAP システムからの技術情報が spec.md 作成に不可欠である。SDD 標準フローには Design → Specify の間にこの情報収集フェーズがなかった。

**決定:**
Gate 1,2 承認後、Specify Phase 前に Phase 1.5 を挿入する。7 ステップ（1.5-A1 ~ 1.5-C2）で SAP システムから技術コンテキストを `sap_context.md` に記録する。

**v2 ステップ構成:**

| Step ID | 内容 | ツール |
|---------|------|--------|
| 1.5-A1 | SAP オブジェクト検索 | search.js |
| 1.5-A2 | SAP ソース参照 | read.js |
| 1.5-A3 | SAP ソース取得 | pull.js（gated） |
| 1.5-B1 | sap_context.md 記録 | AI |
| 1.5-B2 | テーブルメタデータ記録 | sap_context_metadata.py |
| 1.5-C1 | T100 メッセージ検証 | sap_message_t100_validator.py |
| 1.5-C2 | DDIC 存在検証 | sap_ddic_gate_validator.py |

**理由:**
- spec.md の AC を具体化するためにテーブル構造やデータ型の情報が必須
- 事前にテストデータの存在を確認しておくことで、Execute Phase での手戻りを防止
- `pull.js` によるソース取得は人間承認が必要（gated authority）であり、独立したフェーズとして管理が適切

**トレードオフ:**
- フロー全体が 1 フェーズ増えるため、小規模開発では過剰に感じる場合がある
- SAP システム接続が必須のため、オフライン環境では実行不可

## ADR-003: ツール実行と AI 判断の分離

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2026-03 |
| **カテゴリ** | ワークフロー設計原則 |

**背景:**
初期設計では 1 つのステップ内でツール実行と AI による判断・記述が混在していた。これにより、ツールが出力した情報を AI が正しく反映しないケースや、ツール実行をスキップして AI が独自に情報を生成するケースが発生していた。

**決定:**
ステップを「ツール実行（Tool）」と「AI 判断・記述（AI）」に明確に分離する。ツールステップは stdout に結果を出力し、直後の AI ステップがその結果を成果物に反映する。

**理由:**
- ツールの出力が客観的なエビデンスとなり、AI の「でっちあげ」を防止
- ツールの出力を AI が参照するフローが明示されることで、手順遵守の検証が容易
- MANIFEST.yaml の `phase_steps` でツール実行ステップと AI ワークフローステップを `type` で区別

**トレードオフ:**
- ステップ数が増加する（1 ステップが 2 ステップに分割される場合がある）
- ツール出力の AI への受け渡しにコンテキスト消費が発生

**v2 具体例:**
- Step 1.5-B2（ツール: `sap_context_metadata.py` → stdout）→ Step 1.5-B1 で AI が sap_context.md に反映
- Step 2-D1 ~ 2-D4（各 validator がチェック結果を出力）→ AI が検出項目を修正

## ADR-007: T100 メッセージクラスの強制（ハードコードメッセージ禁止）

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2025-11 |
| **カテゴリ** | ABAP コーディング規約 |

**背景:**
ABAP ではメッセージ出力に 2 つの方法がある:
1. `MESSAGE 'テキスト' TYPE 'E'.`（ハードコード）
2. `MESSAGE e001(ZMSG_CLASS) WITH ...`（T100 メッセージクラス参照）

ハードコードメッセージは多言語対応不可、メッセージ番号による管理不可、テストでの検証が困難。

**決定:**
ハードコードテキストメッセージを全面禁止。T100 メッセージクラスの使用を強制する。`sap_common_class_lint.py` が `forbidden_patterns` としてチェックする（CC-MSG-01 ルール）。

**理由:**
- メッセージ番号による追跡が可能（SY-MSGNO で検証可能）
- テストシナリオの `expected_result` でメッセージ番号を指定でき、自動検証が実現
- 多言語対応の基盤

**トレードオフ:**
- 開発初期にメッセージクラスの事前登録が必要
- メッセージクラスの管理コストが増加

## ADR-009: 3 層権限分離モデル

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2025-12（v2 で Execution Authority に再編） |
| **カテゴリ** | セキュリティ / 実行モデル |

**背景:**
AI（Claude Code）に SAP システムへの全アクセス権を与えると、意図しない変更（ソース上書き、本番移送）のリスクがある。一方で、全操作に人間承認を要求するとワークフローが遅延する。

**決定:**
操作を 3 層に分類（v2 では `phase4_execute.md` の Execution Authority テーブルとして定義）:

| 層 | 操作例 | AI の権限 |
|---|--------|----------|
| **conversational**（自律実行） | `search.js`, `read.js`, `data_preview.js`, `activate.js`, `run_tests.js`, lint 系ツール | AI が自律実行 |
| **gated**（人間承認必須） | `pull.js` | 人間承認後に AI 実行 |
| **prohibited**（絶対禁止） | SAP 本番系への直接アクセス、TR の本番リリース、ユーザーマスタ変更 | AI 実行禁止。人間のみ |

**v2 での変更点:**
- v1 の「Rule S3」から `phase4_execute.md` の Execution Authority テーブルに移行
- MANIFEST.yaml の `sap_operations` でツールごとに `authority` レベルを宣言

**理由:**
- 読み取り操作（search/read）は副作用がなく、AI 自律実行が安全
- 書き込み操作（activate）は Gate チェック後なら安全
- 承認操作（APPROVAL.md）は AI が編集すると「自己承認」となり、チェック機能が無効化される

**トレードオフ:**
- pull.js のソース取得にも人間承認が必要であり、Phase 1.5 の速度が制限される
- 層の分類変更には MANIFEST.yaml + CLAUDE_SAP.md の改定が必要

## ADR-010: 共通クラスルールの YAML 管理

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2026-01（v2 で拡張） |
| **カテゴリ** | 設定管理 / 拡張性 |

**背景:**
SAP 開発では共通クラス（ZCL_CMA_*）と共通 Include（ZCMAI*）の使い分けが重要だが、どの AC に対してどの共通部品を使うべきかの判断が開発者（AI 含む）に委ねられていた。

**決定:**
`config/common_class_rules.yaml` に 2 つのセクション（`common_class_rules` + `common_include_rules`）を設け、trigger_pattern（正規表現）で AC や選択画面定義にマッチさせる。マッチ結果は spec.md に反映され、`sap_common_class_lint.py` が実装時の使用を検証する。

**v2 での拡張:**
- v2 では 17 クラス / 92 メソッドに規模が拡大（MANIFEST.yaml の config セクションで管理）
- `forbidden_patterns`（CC-MSG-01 等）による禁止パターンチェックは引き続き有効

**理由:**
- ルールベースの自動判定により、AI の恣意的な共通部品選択を防止
- YAML ファイルの編集だけで新しい共通部品を追加でき、ツールコードの変更が不要
- 禁止パターン（`forbidden_patterns`）により、共通クラスを使わずに標準 API を直接使うケースを検出

**トレードオフ:**
- trigger_pattern の正規表現が複雑になると誤検知・検知漏れのリスク
- 新しい共通部品の追加時に YAML と ABAP 両方のメンテナンスが必要

## ADR-011: S-Evidence の位置づけ（ドメイン用語としての定義）

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2026-03 |
| **カテゴリ** | 用語定義 |

**背景:**
「S-Evidence」という名称がワークフローのステップ名として使われる場面と、成果物の名称として使われる場面が混在していた。

**決定:**
S-Evidence は**成果物名（ドメイン用語）**として定義する。ステップとしては Final Phase のエビデンス収集ステップ（5-A1）で作成される。コードやドキュメントで `S-Evidence` と記述する場合は常に成果物を指し、ステップ参照にはステップ ID を使う。

**理由:**
- 名前空間の衝突を避け、会話やドキュメントでの誤解を防止
- `evidence_capture.js` の出力 HTML のタイトルに S-Evidence を使うのは成果物名として自然
- ステップ ID はワークフローの参照として明確

**トレードオフ:**
- 2 つの名前（S-Evidence = 成果物、ステップ ID = ステップ）を使い分ける必要がある
- 新規参画者にとって直感的でない可能性

## ADR-012: SAP 開発順序の強制（DDIC → CDS → BDEF → CLAS → DCLS → SRVD → SRVB）

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2025-11 |
| **カテゴリ** | SAP 技術制約 |

**背景:**
SAP S/4HANA のオブジェクトには依存関係があり、CDS ビューはテーブルに依存し、Behavior Definition は CDS ビューに依存する。この順序を無視するとアクティベーションエラーが発生する。

**決定:**
tasks.md の Gate Check で SAP 開発順序の整合性を自動検証する（`phase3_tasking.md` に定義）。タスクの依存関係が以下の順序に従っていることを確認:

```
DOMA → DTEL → TABL → DDLS → BDEF → CLAS → DCLS → SRVD → SRVB → DDLX
```

**理由:**
- SAP システムのアクティベーション依存関係に完全に合致
- 順序違反を Gate 5 で検出することで、Execute Phase での手戻りを防止
- AI が生成するタスク分解の品質を機械的に担保

**トレードオフ:**
- 小規模な変更（既存クラスの修正のみ）では過剰な制約となる場合がある
- 新しいオブジェクトタイプ（SAP BTP 拡張等）には順序定義の更新が必要

## ADR-013: テストの TDD 原則（AC 単位の Red-Green）

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2026-02 |
| **カテゴリ** | テスト戦略 |

**背景:**
AI がコード実装とテスト作成を一括で行い、後から全テストを実行する方式では、テスト失敗時のデバッグが困難であった。また、テストが実装に合わせて書かれるため、仕様との乖離を検出できなかった。

**決定:**
AC 単位の TDD（Test-Driven Development）を採用（`phase4_execute.md` Step 6-2 に定義）。1 AC を実装するたびに Red（テスト失敗）→ Green（テスト成功）を確認する。一括実装 → 一括テストは禁止。

**理由:**
- AC 単位でのテストにより、仕様と実装の乖離を即座に検出
- テスト失敗時の原因特定が容易（直前の変更が原因）
- 品質スコア（`sap_quality_score.py`）による最終品質チェックとの整合性

**トレードオフ:**
- SAP システムへの activate → run_tests の往復回数が増加
- 小さな AC が多い場合、開発リズムが細切れになる

## ADR-014: Phase ベースのドキュメント構成

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2026-06 |
| **カテゴリ** | ドキュメント構成 |

**背景:**
v1 では agent_docs がトピック別（CLAUDE_SAP.md に全ルール、CLAUDE_WORKFLOW_SAP.md に全ステップ）に構成されていた。これにより、特定 Phase の作業時に不要な情報まで読み込む必要があり、コンテキスト消費が増大していた。また、1 ファイルが巨大化し、メンテナンスが困難であった。

**決定:**
v2 では agent_docs を Phase 別に分割する。各 Phase に専用のドキュメントを用意し、MANIFEST.yaml の `agent_docs[].load_timing` で読み込みタイミングを制御する:

| ドキュメント | load_timing | 内容 |
|-------------|-------------|------|
| CLAUDE_SAP.md | bootstrap | Rule S1-S7（常時適用） |
| CLAUDE_WORKFLOW_SAP.md | bootstrap | Phase 概要 + 品質基準（常時適用） |
| phase1_design.md | phase:design | yaml→basic_design 転記手順 |
| phase15_sap_context.md | phase:design | SAP 実機接続・コンテキスト記録 |
| phase2_specify.md | phase:specify | spec/plan/contracts 生成 |
| phase3_tasking.md | phase:tasking | SAP 開発順序・mode 判定 |
| phase4_execute.md | phase:execute | WI 実装サイクル・TDD |
| phase_final.md | phase:final | エビデンス・PR チェック |

**理由:**
- Phase ごとに必要な情報のみ読み込むことで、コンテキスト消費を削減
- 各ドキュメントが小さくなり、メンテナンスが容易
- MANIFEST.yaml の `load_timing` により、読み込みルールが機械的に定義される

**トレードオフ:**
- ドキュメント数が増加し、ファイル間の参照関係が複雑になる
- Phase をまたぐ情報（共通ルール等）は bootstrap ドキュメントに集約する必要がある

## ADR-015: MANIFEST.yaml による拡張パックレジストリ

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2026-06 |
| **カテゴリ** | 拡張パック基盤 |

**背景:**
v1 ではツール・バリデーター・テンプレートの一覧が複数のドキュメント（CLAUDE_WORKFLOW_SAP.md、CLAUDE_SAP.md 等）に散在していた。新しいツールの追加や既存ツールの変更時に、複数ファイルの同時更新が必要であり、不整合が発生しやすかった。

**決定:**
`MANIFEST.yaml`（v2.0.0）を拡張パックの単一レジストリとして導入する。以下を一元管理:

| セクション | 管理対象 |
|-----------|---------|
| `extension` | 拡張パックメタデータ（名前・バージョン・対応 STRIDE バージョン） |
| `detection` | 拡張パック有効化の検出条件（3 戦略） |
| `tools.validators` | バリデーター一覧（trigger + severity 付き） |
| `tools.utilities` | ユーティリティツール一覧 |
| `tools.sap_operations` | SAP 操作ツール一覧（authority レベル付き） |
| `tools.mapping_refs` | AI 参照用マッピング定義 |
| `templates` | テンプレート置換定義（substitutions + partial_templates） |
| `agent_docs` | ドキュメント一覧（load_timing 付き） |
| `config` | 設定ファイル一覧 |
| `phase_steps` | Phase 別ステップ定義（ID + ツール + type） |

**理由:**
- ツール・テンプレート・ドキュメントの追加・削除が MANIFEST.yaml の編集のみで完結
- `phase_steps` により、各 Phase のステップ構成が機械可読な形で定義される
- `detection.strategies` により、拡張パックの自動検出が可能
- AI がブートストラップ時に MANIFEST.yaml を読むことで、利用可能なツール・テンプレートを把握できる

**トレードオフ:**
- MANIFEST.yaml 自体が大きくなり、手動編集時のミスリスク（YAML 構文エラー等）
- ツールの実装詳細は MANIFEST.yaml には含まれないため、詳細は個別ドキュメントを参照する必要がある

## ADR-016: 共通クラス正チェックの廃止

| 項目 | 内容 |
|------|------|
| **ステータス** | 採用済み |
| **決定日** | 2026-06 |
| **カテゴリ** | 品質チェック / 実行モデル |

**背景:**
v1 では `sap_common_class_lint.py` が 2 種類のチェックを実行していた:
1. **負チェック（negative）**: 禁止パターンの検出（例: ハードコードメッセージ → CC-MSG-01）
2. **正チェック（positive）**: 共通クラスの使用推奨の確認（例: ZCL_CMA_LOG を使うべき箇所で使っているか）

正チェックは誤検知が多く、AI が lint エラーを解消するために不必要な共通クラス呼び出しを追加するケースが発生していた。

**決定:**
v2 では正チェックを lint スクリプトから削除し、以下の 2 層で代替する:
1. **ワークフロー層**: `phase4_execute.md` Step 6-2 の TDD サイクル内で、AI が共通クラス使用の判断を自律的に行う（`common_class_rules.yaml` を参照）
2. **エビデンス層**: 人間レビューで共通クラスの使用妥当性を確認

**理由:**
- 正チェックの誤検知率が高く、lint 修正のコストが品質向上効果を上回っていた
- 共通クラスの使用判断は文脈依存であり、機械的なパターンマッチでは限界がある
- AI の自律判断 + 人間レビューの 2 層モデルの方が、品質とスループットのバランスが良い
- 負チェック（禁止パターン検出）は引き続き有効であり、これで十分なガードレールとなる

**トレードオフ:**
- 正チェックの廃止により、共通クラスの使い忘れを機械的に検出できなくなる
- AI の判断品質に依存するため、共通クラスの使用ガイドライン（common_class_rules.yaml）の充実が重要
