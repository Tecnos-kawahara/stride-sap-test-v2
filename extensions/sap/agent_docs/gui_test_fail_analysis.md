# GUI テスト FAIL 時の原因分析ガイド

> GUI テスト（Step 6-4 / S1-D4）が FAIL した場合の原因分析手順。
> ツール実行時のエラー（接続エラー、構文エラー等）は `error_recovery_sap.md` を参照。

---

## GUI テストの原則

WSF スクリプトは**ユーザーが画面上で行う操作をそのまま自動化したもの**である。
テストの本質は「ユーザーが画面で操作した場合に正しく動作するか」の検証であり、
WSF はその操作を再現する手段にすぎない。

この原則から以下が導かれる:

- **WSF を弄り回して PASS させることに意味はない。** WSF が再現する操作はユーザーが画面でする操作と同じでなければならない。WSF 側に特殊な回避策（通常のユーザー操作にない SendKeys シーケンス、フィールドの Clear/SetFocus の追加等）を入れて PASS させても、それはユーザーが実際に操作した場合の動作を検証していない
- **標準的なユーザー操作で FAIL するならソース側の問題。** WSF が「値を入力して F8 を押す」という通常のユーザー操作を再現してエラーが出るなら、ユーザーが手で操作しても同じエラーが出る。WSF の書き方ではなく ABAP ソースコードを疑うこと
- **WSF 修正が許されるのは WSF 自体の設定ミスのみ。** CONFIG の値の誤り（存在しない会社コード等）、フィールド名の typo、テストパターンの有効/無効設定の誤り等、WSF 自体の設定に起因する問題のみ WSF 側で修正する

---

## FAIL 時の原因分析手順

テストが FAIL した場合、以下の手順で原因を特定する。推測で修正を重ねることは禁止。

```
1. 個別テスト実行で切り分ける
   --auto での全テスト一括実行で FAIL した場合、まず単一のテストケースを
   個別に実行して問題を再現する。全テスト一括で繰り返し実行しない。
   （複数テストを一括実行すると、どのテストが原因かの切り分けができない）

2. エラーメッセージを正確に読む
   表示されたエラーメッセージの内容を正確に把握する。
   「どうせこれだろう」で推測せず、メッセージが示す意味を文字通りに理解する。

3. ユーザー手動操作で再現確認する（可能な場合）
   WSF と同じ操作を SAP GUI 上で手動実行し、同じエラーが出るか確認する。
   - 同じエラーが出る → ABAP ソース側の問題（WSF は無関係）
   - エラーが出ない → WSF の操作再現が実際のユーザー操作と異なっている可能性

4. ABAP ソースコードを精読する
   エラーの原因に応じて、関連するコードをしっかりと精査する。
   斜め読みで「この行だろう」と即断せず、エラーが発生する処理の流れを
   入口から出口まで追跡し、どの条件でどの分岐に入るかを正確に把握する。

5. デバッグと実装修正を分離する
   原因が ABAP ソースにあると判断した場合でも、いきなり本番修正しない。
   まずハードコードの一時メッセージ等で原因を確定させてから、
   正式な修正（T100 登録、メソッド変更等）を実施する。
```

**禁止事項**:
- エラーの原因分析をせずに WSF の修正を繰り返すこと
- `--auto` で全テストを一括実行し続けて原因切り分けをしないこと
- ABAP ソースを読まずに「実装は正しい」と即断し WSF 側だけを疑うこと

---

## FAIL 後の修正フロー

```
GUI テスト FAIL
  └─ 上記手順で原因特定
       ├─ ABAP ソースの問題 → S1-B1 に戻り修正 → S1-C1 → S1-D1 → S1-D2 → S1-D4
       └─ WSF の設定ミス（CONFIG 値の誤り等に限る） → WSF 修正 → S1-D4 再実行
```
