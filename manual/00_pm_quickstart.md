# PM クイックスタート（5分で理解する Tecnos-STRIDE）

> **対象**: プロジェクトマネージャー、ビジネスオーナー
> **所要時間**: 5分
> **前提知識**: 不要

---

## 30秒で理解する Tecnos-STRIDE

```
あなた: 「受注登録機能を作って」

     ↓  AI が全部やる（設計・仕様・実装・テスト）

AI:  「設計できました。確認して承認してください」 ← ここだけ人間

     ↓  承認する（チェックボックスを入れるだけ）

AI:  「次の工程も終わりました。確認して承認してください」

     ... 繰り返し ...

AI:  「全部完了しました。PR を作成しますか？」
```

**PMがやること**: 各工程の成果物を確認し、APPROVAL.md のチェックボックスを `[x]` にするだけ。
**それ以外は全てAI（Claude Code）が自律実行します。**

---

## 全体フロー（6ステップ）

```
Step 1        Step 2         Step 3         Step 4           Step 5        Step 6
要件を伝える → 基本設計 →    仕様策定 →     タスク分解 →     実装 →        最終確認
              ⛔承認         ⛔承認         ⛔承認          ⛔承認(WI別)   ⛔承認→PR

  あなた      AI→あなた     AI→あなた      AI→あなた       AI→あなた     AI→あなた
```

⛔ = AI が停止して承認を待つポイント（**ここだけ人間の判断が必要**）

---

## あなたがやること（3つだけ）

### ① 要件を伝える

Claude Code に自然言語で伝えるだけです。

```
以下の機能を開発してください:

【機能名】Web-EDI 受注受付
【誰が使う】取引先の購買担当者（約80社）
【何をする】Web-EDI で発注データを受け付け、ERP に自動登録
【なぜ必要】メール/Excel 手入力で誤入力が多い
【関連システム】SAP S/4HANA, Web-EDI ポータル

SDD 開発を開始してください。
```

> 💡 要件が曖昧なら「Intake-First で質問形式で聞き取って」と伝えると、AIが1つずつ対話で整理します。

### ② 成果物を確認する

AI が各工程を完了すると、こう報告します:

```
Gate 1,2 の承認をお願いします。
APPROVAL.md の「Gate 1: Basic Design」「Gate 2: BPMN」セクションで:
  1. 確認項目のチェックボックスを **すべて** [x] に変更
  2. 承認者名と日付を記入
してください。
```

**確認ポイント（工程別）**:

| 工程 | 確認すること | 所要時間目安 |
|------|------------|------------|
| **基本設計** (Gate 1,2) | 目的・対象・業務フローが正しいか | 10分 |
| **仕様** (Gate 3,4) | 受入条件（AC）が業務要件を満たしているか | 15分 |
| **タスク分解** (Gate 5) | 作業項目に抜け漏れがないか | 5分 |
| **実装** (WI 単位) | 動作確認・テスト結果が妥当か | WI あたり 5分 |
| **最終確認** (Final) | Evidence Pack（品質証跡）が揃っているか | 10分 |

### ③ 承認する

`specs/<機能名>/APPROVAL.md` を開き、チェックボックスを編集します:

```markdown
## Gate 1: Basic Design

確認項目：
- [x] basic_design.md の WHO/WHAT/WHY が正しい        ← すべて [x] に
- [x] トレーサビリティ（RQ→US→AC→BPMN）が定義されている
- [x] RACI+ の役割分担が適切である

承認者: 田中太郎                 ← 名前を記入
日付:   2026-03-10               ← 日付を記入
```

> ⚠️ **チェック項目はすべて `[x]` にしてください。** 1つでも未チェックが残ると stride-lint が不承認と判定します。

編集後、Claude Code に「承認しました」と伝えれば、AI が次の工程に進みます。

---

## ❌ よくある間違い

| やりがちなこと | 正しい対応 |
|--------------|-----------|
| AI に細かく指示を出す | 要件だけ伝えればOK。AI が自律的に進めます |
| lint エラーを心配する | AI が自動修正します。APPROVAL_PENDING 以外は放置でOK |
| APPROVAL.md 以外のファイルを編集する | PM は APPROVAL.md だけ触ればOK |
| 承認を急いで中身を見ない | 各 Gate で 5〜15分の確認は品質の要です |
| 要件変更を口頭で伝える | Claude Code に直接伝えてください（仕様に反映されます） |

---

## 承認の判断基準（迷ったらこれを見る）

### Gate 1,2（基本設計・BPMN）
- [ ] 「誰が・何を・なぜ」が明確か
- [ ] 業務フロー（BPMN）が現実の業務と合っているか
- [ ] 関連システムが正しく特定されているか
- [ ] 未解決の質問 (Blocking Questions) が 0 件か

### Gate 3,4（仕様・計画）
- [ ] 受入条件（AC）がテスト可能な具体性で書かれているか
- [ ] 非機能要件（性能・セキュリティ）が含まれているか
- [ ] API 契約（OpenAPI 等）が定義されているか
- [ ] テスト戦略が妥当か

### Gate 5（タスク分解）
- [ ] 全タスクが仕様の AC にトレースされているか
- [ ] 作業の依存関係が正しいか
- [ ] リスクの高いタスク（権限・DB変更等）が特定されているか

### Final（最終確認）
- [ ] 全テストが PASS しているか
- [ ] Evidence Pack（品質証跡）が揃っているか
- [ ] 運用準備（Ops Pack）が整っているか

---

## 用語集（最低限）

| 用語 | 意味 |
|------|------|
| **Gate** | 次の工程に進むための承認チェックポイント |
| **AC (Acceptance Criteria)** | 「この条件を満たしたら完了」の定義 |
| **WI (Work Item)** | 実装の最小単位。1つずつ AI が実装→承認を繰り返す |
| **stride-lint** | 成果物の品質を自動チェックするツール（AI が自動実行） |
| **Evidence Pack** | テスト結果・カバレッジ等の品質証跡セット |
| **Mode** | WI のリスクレベル（autopilot/confirm/validate の3段階） |
| **APPROVAL.md** | 承認を記録するファイル。**AI は編集禁止、人間専用** |

---

## 困ったときは

| 状況 | 対処 |
|------|------|
| AI が止まった | → APPROVAL.md の承認待ちです。確認して承認してください |
| 要件を変更したい | → Claude Code に変更内容を伝えてください（承認済み Gate は再承認が必要） |
| 成果物の意味がわからない | → 「この basic_design.md を説明してください」と AI に聞いてください |
| AI が間違った設計をした | → 「○○は△△が正しいです」と具体的に修正指示を出してください |
| 進捗を確認したい | → 「現在の Gate 状況を教えてください」と聞いてください |
