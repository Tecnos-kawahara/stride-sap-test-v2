# 16. RACI+ — AI時代の責務分担モデル

> Tecnos-STRIDE における役割と責任の定義
> **v4.4 更新**: AI は全作業の「実行者 (R)」。人間は「承認者 (A)」として APPROVAL.md のみ操作。

---

## 1. RACI+ とは

### 1.1 従来の RACI

| 略語 | 意味 | 説明 |
|------|------|------|
| **R** | Responsible | 実行責任者（作業を行う人） |
| **A** | Accountable | 説明責任者（最終決定権を持つ人） |
| **C** | Consulted | 相談先（意見を求められる人） |
| **I** | Informed | 報告先（結果を通知される人） |

### 1.2 RACI+ の追加要素

AI 実装時代には、新たな役割が必要です：

| 役割 | 説明 | 特徴 |
|------|------|------|
| **AI** | AI エージェント | **全作業の実行者 (R)**（v4.4）。init〜実装〜テスト〜Evidence を自律実行。**承認は不可** |
| **CI** | 継続的統合システム | 自動検証（SAST/SCA/Secrets/Coverage/Tests） |

```
┌─────────────────────────────────────────────────────────────────┐
│  RACI+ (v4.4): AI が「実行」、人間が「承認」                    │
│                                                                 │
│  人間 ──依頼──► AI ──全作業を自律実行──► Gate ──承認依頼──► 人間│
│                      │                          ▲              │
│                      └────── CI 自動検証 ────────┘              │
└─────────────────────────────────────────────────────────────────┘
```

> **v4.4 変更点**: 従来は AI が「提案」し Dev が「実装」する分業でした。
> v4.4 からは AI が init〜実装〜テスト〜Evidence まで自律実行し、人間は Gate 承認のみ行います。
> 詳細: [AI自律実行ガイド (v4.4)](15_ai_autonomous_execution_guide.md)

---

## 2. 役割定義

### 2.1 人間の役割

| 役割 | 責任範囲 | 主なタスク |
|------|----------|-----------|
| **PM** | スコープ・合意・リスク・対顧客 | 要件定義、顧客折衝、リスク管理、進捗報告 |
| **Tech Lead (TL)** | 設計品質・技術判断・レビュー | アーキテクチャ決定、コードレビュー、技術課題解決 |
| **Developer (Dev)** | 実装・テスト実装 | コーディング、単体テスト、デバッグ |
| **QA** | テスト戦略・品質ゲート・再現性 | テスト計画、受入テスト、品質メトリクス |
| **Ops** | 輸送・リリース・監視・ハイパーケア | デプロイ、監視設定、障害対応、運用引継 |

### 2.2 システムの役割

| 役割 | 責任範囲 | 主なタスク |
|------|----------|-----------|
| **AI** | **全作業の実行者 (R)**（v4.4） | init, 設計, 仕様, 計画, タスク, 実装, テスト, lint, Evidence 作成を自律実行 |
| **CI** | 自動検証 | テスト実行、静的解析、セキュリティスキャン |

### 2.3 AI の制約（INVIOLABLE）

> **AI は承認行為を行ってはならない**

| 許可される行為（v4.4: 自律実行） | 禁止される行為（不変） |
|--------------------------------|----------------------|
| init〜実装〜テストの全作業の自律実行 | APPROVAL.md の編集 |
| lint エラーの自動修正 | 承認チェックボックスのチェック |
| Evidence Pack の作成 | 最終決定の代行（A は人間のみ） |
| PR Readiness Check の実行 | 業務判断の独断（要件不明時は人間に確認） |

## 3. Gate × 責務マトリクス

### 3.1 Macro Gate（Gate 1〜Final）

| Gate | 主成果物 | R（実行） | A（説明責任） | C（相談） | I（報告） |
|------|----------|----------|--------------|----------|----------|
| **Gate 1** | basic_design.md | PM | PM | TL, QA | Ops, 顧客 |
| **Gate 2** | process.bpmn | PM, TL | PM | QA | Ops, 顧客 |
| **Gate 3** | spec.md | PM | PM | TL, QA | Ops, 顧客 |
| **Gate 4** | plan.md | TL | TL | PM, QA | Ops |
| **Gate 5** | tasks.md + WI初期化 | TL | TL | PM, QA | Ops |
| **Final** | evidence_pack + ops | QA, Ops | PM + TL | Dev | 顧客 |

### 3.2 Micro Run（Work Item 実行）

| Mode | R（実行） | A（説明責任） | C（相談） | I（報告） |
|------|----------|--------------|----------|----------|
| **autopilot** | Dev, AI | TL | QA | PM, Ops |
| **confirm** | Dev, AI | TL | QA, Ops | PM |
| **validate** | Dev, AI | TL, Ops | QA, PM | 顧客 |

> **validate モードでは Ops も A（説明責任）に近い役割**を担います。

### 3.3 承認チェックポイント

| チェックポイント | 承認者 | タイミング |
|-----------------|--------|-----------|
| **Pre-run: design_diff** | TL | validate 実行前 |
| **Pre-run: plan_review** | TL | confirm/validate 実行前 |
| **Post-run: walkthrough_review** | TL | 全モード実行後 |
| **Post-run: ci_pass** | CI（自動） | 全モード実行後 |
| **Post-run: ops_review** | Ops | 全モード実行後 |

## 4. エスカレーションパス

### 4.1 技術的課題

```
Dev → TL → PM（ビジネス影響がある場合）
```

### 4.2 品質課題

```
QA → TL → PM → 顧客（重大な品質問題）
```

### 4.3 運用課題

```
Ops → TL → PM → 顧客（本番影響がある場合）
```

### 4.4 AI 判断の例外

```
AI が判断できない場合:
  AI → Dev（技術的質問）
  AI → TL（設計判断）
  AI → PM（ビジネス判断）
```

## 5. Mode 別の承認フロー

### 5.1 autopilot（低リスク）

```
┌─────────────────────────────────────────────────────────────────┐
│  1. AI/Dev が実装                                               │
│  2. CI が自動検証                                               │
│  3. Dev が walkthrough.md を作成                                │
│  4. TL が walkthrough_review                                    │
│  5. Ops が ops_review                                           │
│  6. state.yaml を更新 → Done                                    │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 confirm（中リスク）

```
┌─────────────────────────────────────────────────────────────────┐
│  1. Dev が WI を作成                                            │
│  2. TL が plan_review（Pre-run 承認）                           │
│  3. AI/Dev が実装                                               │
│  4. CI が自動検証                                               │
│  5. Dev が walkthrough.md を作成                                │
│  6. TL が walkthrough_review                                    │
│  7. Ops が ops_review                                           │
│  8. state.yaml を更新 → Done                                    │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3 validate（高リスク）

```
┌─────────────────────────────────────────────────────────────────┐
│  1. Dev が WI を作成                                            │
│  2. TL が design_diff_review（Pre-run 承認 1）                  │
│  3. TL が plan_review（Pre-run 承認 2）                         │
│  4. AI/Dev が実装                                               │
│  5. CI が自動検証                                               │
│  6. Dev が walkthrough.md + decision_log.md を作成              │
│  7. TL が walkthrough_review                                    │
│  8. Ops が ops_review                                           │
│  9. state.yaml を更新 → Done                                    │
└─────────────────────────────────────────────────────────────────┘
```

## 7. ベストプラクティス

### 7.1 明確な責任分担

- **A（説明責任）は各タスクで1人だけ**
- 責任が曖昧な場合は事前に明確化
- エスカレーションパスを文書化

### 7.2 AI との協業

- AI の出力は必ず人間がレビュー
- AI に承認権限を与えない
- AI の判断根拠を記録

### 7.3 コミュニケーション

- I（報告）先への適切なタイミングでの通知
- C（相談）先への早めの情報共有
- 重要な決定は文書化
