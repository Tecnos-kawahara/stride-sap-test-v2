# Tecnos-STRIDE v5.x → v6.0 Migration Guide

> Version: v6.0.0-tecnos-stride-value (Phase D, FEAT-VALD01) / Last reviewed: 2026-04-30

既存 v5.x プロジェクト (Tecnos-STRIDE v5.0〜v5.4) を v6.0 仕様 (VALUE Upstream Extension 統合) にアップグレードする手順書です。Phase 0 (Discovery / Elicit / Context Modelling) を後付けする必要があるため、状況に応じて **フル適用 / 部分適用 / 対象外** の 3 つの方針から選択します。

## 1. 対象プロジェクトの判定

まず `specs/<feature>/` 配下の現状を確認します。次のコマンドで `basic_design.md` / `process.bpmn` の有無 + Phase の進捗状況を把握できます。

```bash
ls specs/<feature>/
cat specs/<feature>/state/state.yaml | head -10
```

判定基準は以下の 3 つです (どれか 1 つでも該当すれば該当行に進む):

| 状況 | 推奨方針 | 詳細 |
|---|---|---|
| **basic_design.md 存在 + Phase 1 完了済 + 開発継続** | フル適用 (§3) | upstream_migration_helper.py で Phase 0 seed を逆生成 |
| **basic_design.md 存在 + Phase 1 不完全 / BACCM 部分欠如** | 部分適用 (§4) | 欠けている BACCM 軸のみ追加 |
| **PoC / 短命ツール / 廃止予定 / 保守のみ** | 対象外 (§5) | prototype 降格 or 保留 |

判定に迷った場合は、まず **§4 部分適用** から始めて、必要に応じてフル適用へ切り替えるのが現実的です。

## 2. アップグレード方針

### 2.1 フル適用 (推奨: enterprise-erp / saas-integration profile 案件)

- 目的: Phase 0 → Final まで v6.0 仕様で完走できる状態にする
- 工数目安: 2-4 週間 (規模により変動)
- 前提: basic_design.md が存在し、業務専門家のレビューが可能

### 2.2 部分適用 (推奨: 段階移行 / リスクを抑えたい case)

- 目的: BACCM 6 軸のうち欠けている軸のみ補い、警告レベルから ERROR レベルへ昇格しないようにする
- 工数目安: 3-7 日

### 2.3 対象外 (推奨: 短命ツール / 廃止予定)

- 目的: アップグレードコストを払わない判断
- 工数目安: ほぼゼロ (profile を `prototype` に変更するか、保留)

## 3. 手順 A: フル適用

### 3.1 helper による Phase 0 seed 逆生成

`upstream_migration_helper.py` (Phase D で導入、`sdd-templates/tools/upstream_migration_helper.py`) を使い、既存 basic_design.md から Phase 0 yaml seed を逆生成します。**まず dry-run** で内容を確認してください。

```bash
# Step 1: dry-run (stdout に Markdown ドラフト出力、書込なし)
python3 sdd-templates/tools/upstream_migration_helper.py specs/<feature>/

# Step 2: 内容に納得したら --apply で実書込
python3 sdd-templates/tools/upstream_migration_helper.py specs/<feature>/ --apply
```

これにより `specs/<feature>/upstream/phase_0_discovery/` 配下に `business_need.yaml` / `value_canvas.yaml` / `stakeholder_map.yaml` / `context_map.yaml` / `risk_register.yaml` / `change_strategy.yaml` / `goal_tree.yaml` の **seed (種)** が生成されます。各フィールドには「自動抽出可能」「要人間確認」のラベルが付与されます。

CLI 仕様 (CT-CLI-01):

- `feature_dir` (positional): 対象 v5.x プロジェクトの `specs/<feature>/` パス
- `--apply` (bool, デフォルト False): 実書込、なしは dry-run
- `--profile` (str, デフォルト None): 推奨 profile (None なら basic_design.md から自動検出)
- exit code: 0 = 正常 / 1 = `feature_dir` または `basic_design.md` 不在 / 2 = ERROR (パース失敗等)

### 3.2 人間 refinement (必須)

helper はあくまで **seed 生成** であり、完成したものではありません。以下の手順で refinement します:

1. 各 yaml の「要人間確認」ラベル付きフィールドを業務専門家がレビュー
2. 不足部分を追記、誤抽出を修正
3. `stakeholder_map` で社内外の関係者層を 5-7 階層に整える (enterprise-erp の場合)
4. `value_canvas` で「現状 → 目標状態」の差分を明確化

### 3.3 BACCM 完成度確認

`stride upstream validate <feature>` で BACCM 6 軸の完成度をチェックします。

```bash
sdd-templates/bin/stride upstream validate <feature>
```

`baccm_completeness_checker.py` が profile 別の閾値 (`shared/policies/baccm_completeness.yaml`) で評価します。

- **enterprise-erp**: 全軸 100% 必須 (`threshold_for_gate_0: 100`)
- **saas-integration**: 全軸 pass 必須
- **prototype**: stakeholder + value のみ pass、他は warning

## 4. 手順 B: 部分適用

`stride upstream validate <feature>` で fail した BACCM 軸のみを追加します。例えば `change` 軸が fail した場合:

```bash
# 1. specs/<feature>/upstream/phase_0_discovery/ ディレクトリを作成 (ない場合)
mkdir -p specs/<feature>/upstream/phase_0_discovery/

# 2. 不足軸の yaml だけ作成
# 例: change_strategy.yaml だけ書く
cp sdd-templates/templates/upstream/change_strategy.yaml specs/<feature>/upstream/phase_0_discovery/

# 3. 編集して埋める

# 4. 再度 validate
sdd-templates/bin/stride upstream validate <feature>
```

部分適用では `business_need / value_canvas` 等の他軸を空欄のまま残せますが、**enterprise-erp profile では最終的に全軸 100% が必要** なので、Final Gate までに残軸を埋める計画を立ててください。

## 5. 手順 C: 対象外 (prototype 降格 or 保留)

PoC / 短命ツール / 廃止予定の case では、フル適用 / 部分適用は工数対効果が悪いため、以下のいずれかを選びます:

### 5.1 prototype profile に降格

`specs/<feature>/basic_design.md` の `basic_design.profile` を `prototype` に変更し、`state/state.yaml` の `profile` も同期します (SSoT は basic_design.md 側、`stride init` 再実行で同期可)。prototype profile では BACCM 必須は stakeholder + value 2 軸のみのため、他軸は warning のままで運用できます (`manual/50_prototype_value_playbook.md` 参照)。

### 5.2 保留 (移行しない)

v5.x のまま運用継続する判断もあります。ただし v6.0 で導入された `stride upstream-bridge` / `stride retro --solution-eval` 等の新機能は使えません。**廃止予定が明確な場合に限定** してください。

## 6. 検証

アップグレード完了後、以下を確認してください:

```bash
# 1. upstream lint
sdd-templates/bin/stride lint --upstream specs/<feature>/

# 2. BACCM 完成度
sdd-templates/bin/stride upstream validate <feature>

# 3. iteration 評価 (任意)
sdd-templates/bin/stride evaluate specs/<feature>/ --phase discovery

# 4. 全体 lint
sdd-templates/bin/stride lint specs/<feature>/

# 5. テスト実行 (regression 防止)
python3 -m pytest -q
```

すべて PASS していれば v6.0 仕様への移行完了です。Phase 1 (Design) 以降は通常の Tecnos-STRIDE v6.0 SDD フローに従ってください。

## 7. トラブルシューティング (7 項目)

### 7.1 helper 実行時に exit 1 (basic_design.md 不在)

`specs/<feature>/basic_design.md` が見つからないケース。`feature_dir` のパスが正しいか確認するか、`stride init <feature>` で scaffold を再生成してください。

### 7.2 helper 実行時に exit 2 (パース失敗)

`basic_design.md` の YAML frontmatter または canonical YAML 部分の書式エラー。`load_yaml_with_frontmatter(strict=True)` が `yaml.YAMLError` を伝播するため、エラーメッセージで該当行を確認して修正します (`bugfix v7` で導入)。

### 7.3 BACCM 軸が fail し続ける

`stride upstream validate <feature>` の出力で「どの軸の何が不足か」が表示されます。多くの場合、`stakeholder_map.yaml` の役割数不足、`value_canvas.yaml` の現状/目標差分の曖昧さ、`risk_register.yaml` の空項目、が原因です。enterprise-erp で 100% に届かない場合は、複数 iteration で段階的に refinement する想定です。

### 7.4 profile_mismatch warning

`basic_design.md` の `basic_design.profile` と `state/state.yaml` の `profile` 値が異なる場合の警告。SSoT は basic_design.md 側のため、`stride init <feature>` を再実行して state.yaml を同期するか、`state.yaml` を手動で揃えます (`shared/policies/profile_policy.yaml` の `profile_resolution.order` 参照)。

### 7.5 既存 `upstream/` ディレクトリがある場合

`upstream_migration_helper.py --apply` は既存ファイルを **skip + warn** します (冪等性確保)。上書きが必要な場合は、対象ファイルをバックアップしてから手動削除し、再度 `--apply` してください。

### 7.6 `stride upstream-bridge --apply` が Gate 1/2 後で失敗

Phase 1 immutability (Phase C で導入) により、Gate 1/2 承認後の basic_design.md には `--apply` での自動書込ができません。バックアップから書き戻すか、手動で `links` に追記してください (`manual/45_upstream_bridge_guide.md §2.3` 参照)。

### 7.7 移行後のテストで regression が出る

`upstream/` 配下を追加しただけで既存テスト (`tests/test_*.py`) が壊れることは原則ありません。失敗が発生する場合は (a) 既存ファイル改変が起きていないか `git diff` で確認、(b) `python3 -m pytest -q` の失敗 test 名から原因特定、(c) 必要に応じて `--apply` で書き込んだ yaml の不整合を確認してください。

> 移行手順についての追加質問は `agent_docs/sdd_bootstrap.md` または各 Phase のマニュアル (`manual/43-50`) を参照してください。
