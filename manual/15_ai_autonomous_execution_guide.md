# 15. AI自律実行ガイド

> **Version**: v5.4.0-tecnos-stride
> **対象**: Claude Code を使って SDD 開発を行うすべての開発者・PM
> **所要時間**: 約15分

---

## PM向け3分サマリー

**v4.4 で何が変わったか**:

v4.4 以前は、AI は「提案者」であり、人間が逐一指示を出して開発を進めていました。
v4.4 からは、**AI（Claude Code）が全作業の実行者 (R: Responsible)** となり、人間は **承認者 (A: Accountable)** として APPROVAL.md の編集のみ行います。

```
[v4.3以前] 人間が指示 → AI が提案 → 人間が確認 → AI が実行 → 人間が確認 → ...
[v4.4]     人間が依頼 → AI が全フェーズを自律実行 → Gate で停止 → 人間が承認 → AI が再開
```

**PM が行うこと**:
1. Claude Code に「何を作りたいか」を伝える
2. Gate で停止したら成果物を確認し、APPROVAL.md のチェックボックスを編集
3. 業務判断が必要な質問に回答

**PM が行わないこと**:
- コマンド実行、lint、テスト、ファイル作成 — すべて AI が自律実行

## 15.1 AI自律実行モデルとは

### 基本原則

| 役割 | 担当 | 行うこと |
|------|------|----------|
| **実行者 (R)** | Claude Code | init, 設計, 仕様, 計画, タスク, 実装, テスト, lint, Evidence 作成 |
| **承認者 (A)** | 人間 | APPROVAL.md の編集、業務判断、要件の決定 |

### v4.3 以前との比較

| 観点 | v4.3 以前 | v4.4 |
|------|----------|------|
| AI の役割 | 提案・下書き・支援 | **全作業の実行者** |
| 人間の役割 | 指示・確認・実行 | **承認のみ** |
| Phase 内の進行 | 都度人間が確認・指示 | **AI が連続実行**（Gate でのみ停止） |
| lint エラー | 人間に報告して待機 | **AI が自動修正** |
| テスト実行 | 人間の指示で実行 | **AI が自律実行** |
| 停止条件 | 各ステップで停止 | **APPROVAL_PENDING のみ** |

### AI の停止条件（2つだけ）

Claude Code が停止するのは、以下の2つの場合**のみ**です:

| 停止条件 | トリガー | 人間の対応 |
|----------|---------|-----------|
| **APPROVAL_PENDING** | Gate 1〜5, Final の承認待ち | APPROVAL.md のチェックボックスを `[x]` に |
| **WI_APPROVAL_PENDING** | Work Item の承認待ち | WI-*.approval.md のチェックボックスを `[x]` に |

それ以外のエラー（lint FAIL、テスト失敗、プレースホルダ残留等）は AI が自動で修正します。

## 15.3 実践: 使い方ガイド

### Step 1: プロジェクトの準備

プロジェクトに SDD テンプレートが配置されていることを確認します。

```bash
# 方法1: クローン
git clone https://github.com/tecnos-japan-cbp/tecnos-sdd-template-enterprise.git my_project
cd my_project

# 方法2: 既存プロジェクトにコピー
cp -r /path/to/sdd-templates ./
cp -r /path/to/agent_docs ./
cp -r /path/to/memory ./
cp /path/to/CLAUDE.md ./
cp /path/to/CLAUDE_WORKFLOW.md ./
cp /path/to/SDD_MANIFESTO.md ./
```

### Step 2: Claude Code を起動して開発を開始

Claude Code はプロジェクト内の `CLAUDE.md` を自動で読み込みます。
`CLAUDE.md` が `agent_docs/sdd_bootstrap.md` の読み込みを指示するため、
Claude Code は **1ファイルを読むだけで全ルールを理解**し、すぐに開発を開始できます。

> **Note**: 以前のバージョンでは7つ以上のファイルを事前に読む必要がありましたが、
> v4.4 では `sdd_bootstrap.md` に統合されたため、オンボーディングが大幅に高速化しました。

プロジェクトフォルダで Claude Code を起動し、要件を伝えるだけです。

**方法A: 要件を一度に伝える（最速）**

```
以下の機能を開発したいです:

【機能名】Web-EDI受注受付
【誰が使う】取引先の購買担当者（約80社）
【何をする】Web-EDIで発注データを受け付け、ERPに自動登録する
【なぜ必要】現状はメール/Excelで受注を手入力しており、誤入力が多い
【関連システム】SAP S/4HANA, Web-EDIポータル

SDD開発を開始してください。
```

**方法B: Intake 対話モード（要件が曖昧な場合に推奨）**

```
Intake-First で SDD 開発を始めてください。
機能名は「web_edi_order」です。
質問形式で要件を聞き取ってください。
```

Claude Code が Who/What/Why、Scope、関連システム、業務フロー、制約を**1つずつ対話で質問**し、回答をもとに `basic_design_intake.md` を自動記入した後、`basic_design.md` を生成します。

### Step 3: AI が自律的に作業を進める

Claude Code は以下を**指示なしで連続実行**します:

1. `stride init web_edi_order --detect` でディレクトリ構造を作成
2. `basic_design.md` を作成（要件から Who/What/Why を構造化）
3. `process.bpmn` を作成（業務フローを可視化）
4. `stride lint` を実行し、エラーがあれば自動修正
5. lint PASS を確認

そして以下のように報告します:

```
Design Phase が完了しました。

Gate 1,2 の承認をお願いします。
APPROVAL.md の「Gate 1: Basic Design」と「Gate 2: BPMN」セクションで:
  1. チェックボックスを [x] に変更
  2. 承認者名と日付を記入
してください。
```

### Step 4: 人間が承認する

`specs/web_edi_order/APPROVAL.md` を開き、チェックボックスを編集します:

```markdown
## Gate 1: Basic Design
- [x] basic_design approved
- Approver: 田中太郎
- Date: 2026-02-15
```

承認後、Claude Code に「承認しました」と伝えるだけで、次の Phase に自動的に進みます。

### Step 5: 以降も同じパターン

Specify → Tasking → Execute → Final まで、同じパターンの繰り返しです:

1. AI が自律的に作業を実行
2. Gate で停止して承認を依頼
3. 人間が APPROVAL.md を編集
4. 「承認しました」と伝える
5. AI が次の Phase に進む

## 15.5 AI の自動修正範囲

### AI が自動修正するもの

| カテゴリ | 例 |
|----------|-----|
| **構造エラー** | 必須セクション欠落、YAML 構造不正 |
| **カウント不一致** | AC 数と counts の不一致 |
| **プレースホルダ残留** | `XXX`, `NNN`, `TODO` の置換漏れ |
| **テスト失敗** | テストコードの修正、再実行 |
| **lint WARN** | 非クリティカルな警告の修正 |

### AI が修正しないもの（人間の対応が必要）

| カテゴリ | 対応方法 |
|----------|---------|
| **APPROVAL_PENDING** | APPROVAL.md のチェックボックスを `[x]` に |
| **WI_APPROVAL_PENDING** | WI-*.approval.md のチェックボックスを `[x]` に |
| **業務判断** | Claude Code の質問に回答（自然言語で） |
| **要件不明** | 不明点を解消してから「続けてください」 |

## 15.6 RACI+ における v4.4 の変更

### v4.4 での AI の役割変化

| 観点 | v4.3 以前 | v4.4 |
|------|----------|------|
| RACI での位置 | 補助的な R（提案・支援） | **全作業の R（実行者）** |
| 承認 (A) | 人間のみ ← 変更なし | 人間のみ ← 変更なし |
| 不変ルール | AI は A になれない ← 変更なし | AI は A になれない ← 変更なし |

### APPROVAL.md ルール（不変）

v4.4 でも以下のルールは**絶対に変更されません**:

- **AI は APPROVAL.md を編集できない**
- **承認チェックボックスは人間のみが操作する**
- **AI は Accountable（最終責任者）になれない**

これは安全装置であり、AI がどれだけ自律的に作業を行っても、最終的な品質の判断と責任は人間にあります。

## 15.7 トラブルシューティング

| 問題 | 原因 | 解決策 |
|------|------|--------|
| AI が途中で停止した | APPROVAL_PENDING | APPROVAL.md を編集して「承認しました」と伝える |
| AI が先に進まない | 前の Gate が未承認 | 該当 Gate の承認を確認 |
| AI が lint エラーを修正できない | 業務判断が必要 | エラー内容を確認し、判断を伝える |
| AI が間違った修正をした | 要件の解釈違い | 「○○は△△が正しいです」と修正を指示 |
| AI がルールを無視している | コンテキスト消失 | 「CLAUDE.md を読み直してください」と伝える |
| Gate 承認後に変更が必要 | 要件変更 | 「○○を変更してください。再承認します」と伝える |

## 15.8 ベストプラクティス

### 人間側のベストプラクティス

1. **要件は最初にしっかり伝える** — AI は要件をもとに全フェーズを自律実行するため、最初の要件の質が成果物の質を決めます
2. **Gate 承認は成果物を確認してから** — AI が停止して報告した内容を確認し、問題なければ承認
3. **業務判断は迅速に** — AI は判断待ちで停止するため、迅速な回答が開発速度を左右します
4. **修正依頼は具体的に** — 「もっと良くして」ではなく「○○を△△に変更して」と具体的に

### よくある質問

**Q: AI に任せて品質は大丈夫？**
A: AI は全作業の後に stride-lint で自動検証し、テストも自律実行します。さらに Gate で人間が承認するため、品質は二重に担保されます。

**Q: AI が暴走しないか？**
A: APPROVAL.md の編集は AI に禁止されているため、人間が承認しない限り次の Phase に進めません。これが安全装置です。

**Q: 途中で要件を変更できるか？**
A: できます。変更を伝えれば AI が仕様を更新し、lint で整合性を再検証します。ただし承認済み Gate の変更は再承認が必要です。

**Q: 複数の機能を並行開発できるか？**
A: 可能です。各機能は `specs/<feature>/` で独立しているため、Claude Code に「次は○○の開発を進めてください」と切り替えられます。

**Q: Claude Code 以外のツール（Cursor, Copilot）でも使えるか？**
A: SDD_MANIFESTO.md の Execution Model は全ツール共通です。ただし CLAUDE_WORKFLOW.md の自動実行ルールは Claude Code 固有の設定です。
