## 概要

`specs.md` は、AIネイティブなソフトウェア開発のために設計された先進的なフレームワークです。このフレームワークは、プラグイン可能なフローを提供し、開発のあらゆるユースケースに対応します。Claude Code、Cursor、GitHub Copilotといった主要なAIコーディングツールとシームレスに連携し、マークダウンベースのエージェントを通じて開発プロセス全体を体系的にガイドします。

### フレームワークの核心

`specs.md` は、プロジェクトの規模、複雑さ、そしてチームの要求に応じて選択可能な3つの異なる開発フローを提供します。これにより、開発チームは自身のプロジェクトに最適なレベルの厳密さと構造を適用できます。

**Simple Flow** は仕様生成に特化した軽量なワークフローです。プロトタイプや小規模なタスクに最適であり、実行追跡を必要としない場合に選択します。

**FIRE Flow** は適応型の実行機能と、既存のコードベース（ブラウンフィールド）やモノリシックリポジトリ（モノレポ）への高度なサポートを提供します。作業の複雑さに応じてプロセスの厳密さを動的に調整します。

**AI-DLC Flow** は包括的なトレーサビリティと厳格なプロセス管理を備えた、完全な開発方法論です。AWSが提唱したAIネイティブなソフトウェア開発ライフサイクルに基づいています。

### サポートされるAIコーディングツール

| ツール | サポート方式 |
| :--- | :--- |
| Claude Code | スラッシュコマンドによる完全サポート |
| Cursor | ルールによる完全サポート |
| GitHub Copilot | エージェントによる完全サポート |
| Google Antigravity | エージェントによる完全サポート |
| その他のAIアシスタント | マークダウンプロンプトを読み取れるツールで動作 |

### ライセンス

`specs.md` はMITライセンスの下でオープンソースとして提供されており、個人および商用プロジェクトで自由に利用できます。利用者は、基盤となるAIモデルのAPI利用料のみを負担します。

## インストールとセットアップ

### 基本インストール

`specs.md` の導入は、単一のコマンドで簡単に行うことができます。プロジェクトのルートディレクトリで以下のコマンドを実行することで、必要な設定が自動的に行われます。

```bash
npx specsmd@latest install
```

インストーラーは、開発環境に存在するAIコーディングツールを自動的に検出し、最適な設定を適用します。インストール時にどのフローを使用するかを選択できます。

### グローバルインストールについて

グローバルインストールは推奨されていません。`npx` を使用することで、常に最新バージョンを使用でき、プロジェクト間の競合を避けることができます。

## 開発フローの選択

`specs.md` の最大の強みは、プロジェクトのニーズに合わせて開発フローを選択できる柔軟性にあります。

### クイック決定ガイド

| 質問 | 推奨フロー |
| :--- | :--- |
| 実行追跡なしで仕様だけが必要？ | **Simple Flow** |
| 複雑さに応じた適応的な厳密性が欲しい？ | **FIRE Flow** |
| 包括的なドキュメントと固定ワークフローが必要？ | **AI-DLC Flow** |

### フロー比較表

| 特徴 | Simple Flow | FIRE Flow | AI-DLC Flow |
| :--- | :--- | :--- | :--- |
| **仕様生成** | ✓ | ✓ | ✓ |
| **実行追跡** | - | ✓ | ✓ |
| **ブラウンフィールド対応** | - | ✓ | - |
| **モノレポ対応** | - | ✓ | - |
| **ドメイン駆動設計 (DDD)** | - | - | ✓ |
| **チェックポイント** | なし | 適応型（0-2） | 固定（10-26） |
| **学習曲線** | 低 | 中 | 高 |
| **エージェント数** | 1 | 2 | 4 |

### 推奨ユースケース

**Simple Flow** は、個人プロジェクト、プロトタイプ、小規模なタスクに最適です。仕様を生成するだけで、実行の追跡は必要ない場合に選択してください。他のチームへのハンドオフドキュメントの作成にも適しています。

**FIRE Flow** は、既存のコードベースでの作業、モノレポ、または複雑さに応じて監視レベルを調整したい場合に最適です。ブラウンフィールドプロジェクトを第一級市民として扱い、不要な儀式を排除します。UIのバグ修正と決済ゲートウェイのリファクタリングを同じ厳密さで扱う必要はないという哲学に基づいています。

**AI-DLC Flow** は、複雑なドメインロジック、大規模チーム、または規制環境で広範なドキュメントが必要な場合に最適です。医療、金融、政府機関など、包括的なトレーサビリティが求められる環境で威力を発揮します。

## Simple Flow

### 概要

Simple Flowは、`specs.md` の中で最も軽量かつシンプルなフローです。主に仕様生成に焦点を当てており、迅速なプロトタイピングや小規模なタスクに適しています。Kiroスタイルのスペック駆動開発を、あらゆるAIコーディングツールで実現します。

### 3つのフェーズ

Simple Flowは、直線的で理解しやすい3つのフェーズで構成されています。

#### Phase 1: Requirements（要件）

このフェーズでは、高レベルの目標を具体的なユーザーストーリーと受け入れ基準に変換します。AIエージェントが対話を通じて要件を明確化し、`requirements.md` を生成します。EARS（Easy Approach to Requirements Syntax）基準を使用して、明確で検証可能な要件を定義します。

**出力**: `requirements.md` - ユーザーストーリーとEARS基準を含む要件定義書

#### Phase 2: Design（設計）

技術的なアーキテクチャと設計上の決定を文書化します。コンポーネントの構造、データフロー、APIインターフェースなどが定義されます。Mermaidダイアグラムを使用してアーキテクチャを視覚化します。

**出力**: `design.md` - アーキテクチャとMermaidダイアグラムを含む技術設計書

#### Phase 3: Tasks（タスク）

設計を基に、実装可能な具体的なタスクリストを作成します。各タスクには明確なスコープと完了条件が設定されます。

**出力**: `tasks.md` - コーディングタスクを含む実装チェックリスト

### Simple Flow Agent

このフローは、単一の `Simple Flow Agent` によってガイドされます。

#### 起動方法

```
/specsmd-simple-flow
```

または

```
/specsmd-agent
```

#### コマンド一覧

| コマンド | 説明 |
| :--- | :--- |
| `start` | 新しい仕様の作成を開始します |
| `requirements` | 要件定義フェーズを実行します |
| `design` | 設計フェーズを実行します |
| `tasks` | タスク分解フェーズを実行します |
| `status` | 現在の進捗状況を確認します |

### 核心原則

**Generate First, Ask Later（まず生成、質問は後）**: エージェントは、機能のアイデアに基づいてすぐにドラフトドキュメントを生成します。これは、事前に広範なQ&Aを必要とするのではなく、議論の出発点として機能します。

**Explicit Approval Gates（明示的な承認ゲート）**: 各フェーズを進める前に、明示的に承認する必要があります。「yes」、「approved」、「looks good」と言って続行します。フィードバックがあれば修正がトリガーされます。

**One Phase at a Time（一度に1フェーズ）**: エージェントは1回のインタラクションで1つのドキュメントに集中します。次に進む前に各フェーズを完了させてください。

**One Task at a Time（一度に1タスク）**: 実行中は、1回のインタラクションで1つのタスクのみが実装されます。これにより、各変更を慎重にレビューできます。

### 出力構造

Simple Flowはシンプルでフラットな構造を作成します。

```
specs/
└── {feature-name}/
    ├── requirements.md    # 何を構築するか
    ├── design.md          # どのように構築するか
    └── tasks.md           # ステップバイステップの計画
```

複雑なmemory-bank階層はありません。純粋に仕様のみです。

## FIRE Flow

### 概要

FIRE（Fast Intent-Run Engineering）Flowは、適応性を重視したスペック駆動開発フローです。作業の複雑さに応じてプロセスの厳密さを動的に調整し、特に既存のコードベースでの開発に強みを発揮します。

FIREの哲学は「必要なときにチェック、すべてをチェックするのではない」です。UIのバグ修正と決済ゲートウェイのリファクタリングを同じ厳密さで扱う必要はありません。

### 主な差別化要因

| 特徴 | 説明 |
| :--- | :--- |
| **動的実行** | 事前に計画された実行はありません。Builderが残りの作業項目をスキャンし、依存関係と現在の状態に基づいて次に実行するものを推奨します |
| **適応型チェックポイント** | 複雑さに基づいて作業項目ごとに0-2のチェックポイント。シンプルなタスクは自動操縦、重要な変更は検証 |
| **ファーストクラスのブラウンフィールド** | 既存のパターンを自動検出し、規約を尊重し、コードベースを書き換えるのではなく拡張します |
| **モノレポサポート** | モジュール固有のオーバーライドを持つ階層的な標準。1つのプロジェクト、複数の技術スタック |
| **ウォークスルー生成** | すべての変更が自動的に文書化されます。コードを掘り下げることなくAIが何をしたかをレビュー |
| **変更に優しい** | 要件が変わった？仕様を更新するだけ。再作成する実行計画はありません—次の実行は自動的に適応します |

### フロー構造

```
Intent → Work Items → Runs → Done
```

### 適応型実行モード

FIRE Flowは、タスクの性質に応じて3つの実行モードを提供します。FIREは2つの入力を分析して実行パスを決定します：作業の複雑さと、あなたの自律性設定です。

#### Yolo Mode（自律モード）

Yoloモードは最大の自律性を提供します。AIが計画を立て、実行し、テストを行い、完了時にのみ報告します。シンプルで明確なタスクに最適です。

**特徴:**
- 最小限の中断
- チェックポイント: 0
- 完了時のウォークスルー生成
- 単純なバグ修正やリファクタリングに最適

#### Guided Mode（ガイドモード）

Guidedモードはバランスの取れたアプローチを提供します。AIが計画を提示し、承認後に実行します。中程度の複雑さのタスクに最適です。

**特徴:**
- 実行前の計画レビュー
- チェックポイント: 1
- 主要な決定ポイントでのチェックイン
- 新機能開発に最適

#### Strict Mode（厳格モード）

Strictモードは最大の監視を提供します。各ステップで人間の承認が必要です。複雑なタスクや重要な変更に最適です。

**特徴:**
- 各フェーズでの承認が必要
- チェックポイント: 2
- 詳細な設計ドキュメント
- アーキテクチャ変更に最適

### 核心概念

#### Intent（インテント）

Intentは、ユーザー価値を提供する高レベルの目標です。長い仕様書ではなく、ガイド付きの会話を通じてキャプチャされます。

```yaml
intent:
  id: auth-system
  title: User Authentication System
  status: in_progress
  priority: high
```

#### Work Item（作業項目）

Work Itemは、Intentから派生した個別の実行可能な作業単位です。各作業項目は単一のRunで完了します。

```yaml
work_item:
  id: create-user-schema
  title: Create user database schema
  complexity: low
  mode: autopilot  # 0 checkpoints
```

| フィールド | 説明 |
| :--- | :--- |
| ID | 一意の識別子 |
| Title | 作業項目のタイトル |
| Status | 状態（pending, active, done, blocked） |
| Complexity | 複雑さ（low, medium, high） |
| Mode | 実行モード（autopilot, guided, strict） |
| Dependencies | 依存関係 |

#### Run（実行）

Runは、作業項目の単一の実行サイクルです。コンテキストをロードし、モードに従って実行し、変更を追跡し、ウォークスルーを生成します。

各Runは以下のフェーズを経て進行します：

```
Plan → Execute → Test → Review
```

#### Walkthrough（ウォークスルー）

ウォークスルーは、各Runの後に生成されます。何が変わったか、なぜ変わったか、どのように検証するかを文書化します。

### モノレポとブラウンフィールドへの対応

#### ブラウンフィールドサポート

FIRE Flowは、既存のコードベースを第一級市民として扱います。

**自動検出**: 既存のコードベースをスキャンして技術スタックやコーディング規約を自動的に推測します。

**パターンの尊重**: 既存のパターンを尊重し、コードベースを書き換えるのではなく拡張します。

**最小限の変更**: 必要な変更のみを行い、既存のコードへの影響を最小限に抑えます。

#### モノレポサポート

FIRE Flowは、モノレポを第一級市民としてサポートします。プロジェクトルートの標準設定と、各モジュール固有の設定を階層的に組み合わせることで、大規模なモノレポを効率的に管理します。

```
project-root/
├── .specs-fire/
│   ├── standards/           # ルートレベルの標準
│   │   ├── tech-stack.md
│   │   └── coding-standards.md
│   └── constitution.md      # 普遍的なポリシー（常に継承）
├── packages/
│   ├── frontend/
│   │   └── .specs-fire/
│   │       └── standards/   # フロントエンド固有の標準
│   └── backend/
│       └── .specs-fire/
│           └── standards/   # バックエンド固有の標準
```

#### Constitution（憲法）

Constitutionは、常に継承され、決してオーバーライドされない普遍的なポリシーを定義します。セキュリティルール、コンプライアンス要件、組織全体のガイドラインなどを含みます。

### FIRE Agents

FIRE Flowは2つのエージェントで構成されます。

#### Orchestrator Agent（オーケストレーターエージェント）

Orchestrator Agentは、FIRE Flowの中央コントローラーです。フロー全体を管理し、インテントの作成や実行の開始を担います。

```
/specsmd-fire
```

または

```
/specsmd-fire-planner
```

| コマンド | 説明 |
| :--- | :--- |
| `init` | プロジェクトを初期化 |
| `intent` | 新しいインテントを作成 |
| `work` | 作業項目を作成 |
| `start` | 実行を開始 |
| `status` | 状態を確認 |
| `continue` | 実行を継続 |

#### Worker Agent（ワーカーエージェント）

Worker Agentは、実際の作業を実行するエージェントです。計画、実装、テストといった実際の作業を実行します。

```
/specsmd-fire-worker
```

または

```
/specsmd-fire-builder
```

| コマンド | 説明 |
| :--- | :--- |
| `plan` | 計画を作成 |
| `execute` | 実装を実行 |
| `test` | テストを実行 |
| `review` | レビューを生成 |

### FIRE vs AI-DLC

FIREとAI-DLCは、インテント駆動の哲学を共有していますが、異なるフローです。FIREは適応型実行を優先し、AI-DLCは包括的なトレーサビリティを優先します。

**動的 vs 事前計画された実行**:

- **FIRE（動的）**: Builderエージェントを呼び出すと、残りのすべてのインテントと作業項目をスキャンし、依存関係を分析し、次に一緒に実行できる作業項目を推奨します。事前に実行計画は作成されません。

- **AI-DLC（事前計画）**: Inceptionフェーズで、特定のステージでストーリーをバンドルする事前定義された実行計画である「ボルト」を作成します。ボルトは構築が始まる前に計画されます。

**要件変更への対応**:

AI-DLCでは、ボルト計画後に新しいストーリーを追加したりユニットを変更したりすると、ボルトを再計画する必要がある場合があります。FIREでは、仕様を更新するだけで、Builderは次の実行で変更を確認し、それに応じて推奨します。

## AI-DLC Flow

### 概要

AI-DLC（AI-Driven Development Lifecycle）は、AWSによって提唱された、AIを開発プロセスの中心に据えた包括的な方法論です。AIネイティブな時代のために一から設計されており、従来の方法論にAIを後付けするのではなく、AIを中心に据えて再構築されています。

AI-DLCの核心は「AIが提案し、人間が検証する」という「会話の逆転」です。これはGoogle Mapsのようなものです—人間が目的地を設定すると、AIがステップバイステップの指示を提供し、人間は各チェックポイントで監視を維持します。

### フロー構造

```
Intent → Units → Stories → Bolts → Done
```

### 主な差別化要因

| 特徴 | 説明 |
| :--- | :--- |
| **完全な方法論** | 単なるツールではなく、AWSの研究に基づいた定義されたフェーズ、儀式、成果物を持つ完全なSDLC |
| **DDD統合** | ドメイン駆動設計はオプションではなく、コアです。コードを書く前にビジネスロジックをモデル化 |
| **包括的なチェックポイント** | ボルトごとに10-26の人間検証ポイント。すべての決定がレビューされます |
| **4つの専門エージェント** | Master、Inception、Construction、Operationsエージェントが明確な責任を持つ |
| **ウォークスルー生成** | すべての変更が自動的に文書化されます |

### AI-DLCとは何か

AI-DLCは、AIネイティブな時代のために一から設計された方法論です。

#### 核心原則

**会話の逆転**: 従来の開発では人間がAIに指示を出しますが、AI-DLCではAIが提案し、人間が検証・承認します。

**DDDの統合**: ドメイン駆動設計（DDD）はオプションではなく、方法論の中核に組み込まれています。複雑なビジネスロジックを適切にモデル化するために不可欠です。

**V-Bounceモデル**: 人間は主要な実装者から検証者・確認者へとシフトします。AIが計画、コーディング、テストを担当し、人間は要件、アーキテクチャ、継続的な検証に集中します。

### AI-DLC vs アジャイル

従来のアジャイル手法は、より長いイテレーション期間（週や月）を前提に設計されていました。AIの登場により、イテレーションサイクルは時間や日単位で測定されるようになりました。

| 側面 | アジャイル | AI-DLC |
| :--- | :--- | :--- |
| **イテレーション期間** | 2週間のスプリント | 時間〜日単位のBolt |
| **見積もり** | ストーリーポイント | 複雑さベース |
| **会話の方向** | 人間 → AI | AI → 人間 |
| **主要な実装者** | 人間 | AI |
| **人間の役割** | 実装 | 検証 |

### 3つの開発フェーズ

AI-DLC Flowは3つの主要フェーズで構成されています。

#### Phase 1: Inception（構想）

Inceptionフェーズでは、高レベルの目標（Intent）を捉え、詳細な要件に展開し、管理可能なUnit（ユニット）に分解します。

**ワークフロー**: Capture → Elaborate → Decompose → Plan Bolts

**主な活動:**
- Intentの作成と精緻化
- 要件の詳細化
- システムコンテキストの定義
- Unitへの分解
- ユーザーストーリーの作成
- Bolt計画の策定

**ヒューマンチェックポイント:**
1. Gate 1: 明確化質問への回答
2. Gate 2: 生成された要件の承認
3. Gate 3: 自動生成されたアーティファクトの承認
4. Gate 4: Construction準備完了の確認

#### Phase 2: Construction（構築）

Constructionフェーズでは、Boltを通じて検証済みのステージを経て、テスト済みの本番対応コードを生成します。

**ワークフロー**: Model → Design → ADR → Implement → Test

**主な活動:**
- ドメインモデリング
- 技術設計
- ADR（アーキテクチャ決定記録）の作成
- 実装
- テスト

**ヒューマンチェックポイント:**
各ステージの終了時に、以下のアクションが可能です：
- **Approve**: 次のステージに進む
- **Request Changes**: 修正を依頼
- **Reject**: 前のステージに戻る

#### Phase 3: Operations（運用）

Operationsフェーズでは、構築された機能を本番環境にデプロイし、正しく動作することを確認し、監視を設定します。

> **注意**: Operations Agentは現在アルファ版であり、一部のコマンドが期待通りに動作しない場合があります。

**ワークフロー**: Build → Deploy → Verify → Monitor

**主な活動:**
- ビルドとアーティファクト作成
- 環境へのデプロイ
- 検証
- 監視設定

**ヒューマンチェックポイント:**
1. Gate 1: ビルドアーティファクトの承認
2. Gate 2: ステージングデプロイの検証
3. Gate 3: 本番デプロイの承認
4. Gate 4: 監視設定の完了

**生成されるRunbook:**
- Deployment Runbook: ステップバイステップのデプロイ手順
- Incident Response: 問題発生時の対応手順
- Scaling Runbook: 負荷増加への対応方法

### 核心概念

#### Intents（インテント）

Intentsは、AI-DLCにおける最上位の作業単位です。ユーザーの高レベルな目標を表し、複数のUnitに分解されます。すべての作業の出発点となる高レベルのビジネス目標です。

**Intentの構造:**
```
memory-bank/intents/{intent-name}/
├── intent.md           # インテントの説明
├── requirements.md     # 詳細な要件
├── context.md          # システムコンテキスト
└── units/              # 分解されたユニット
```

#### Units（ユニット）

Unitsは、Intentから派生した凝集性のある作業要素です。各Unitは明確な境界と責任を持ち、独立して実装可能です。疎結合のモジュールとして独立して開発できます。

**Unitの特徴:**
- 単一の責任
- 明確なインターフェース
- 独立したテスト可能性
- 他のUnitとの明確な依存関係

#### Stories（ストーリー）

Storiesは、Unit内のユーザーストーリーで、受け入れ基準を持ちます。

#### Bolts（ボルト）

Boltsは、AI-DLCにおける実行の基本単位です。時間制限のある集中的な作業セッション（時間または日単位）であり、検証済みのステージを経てストーリーを実装します。

**Boltタイプ:**

| タイプ | ステージ数 | 用途 |
| :--- | :--- | :--- |
| DDD Construction | 5 | 複雑なドメインロジック |
| Simple Construction | 3 | UI、統合、ユーティリティ |

**DDD Construction Boltのステージ:**
1. **Domain Model（ドメインモデル）**: アグリゲート、エンティティ、値オブジェクト、ドメインイベント、ユビキタス言語を特定
2. **Technical Design（技術設計）**: パターンを選択、インターフェースを定義、データ構造を計画
3. **ADR Analysis（ADR分析）** - オプション: コンテキスト、オプション、決定、根拠を文書化
4. **Implement（実装）**: 標準に従って本番コードを生成
5. **Test（テスト）**: ユニット、統合、受け入れテストを作成・実行

**Simple Construction Boltのステージ:**
1. **Spec（仕様）**: 要件と設計の確認
2. **Implement（実装）**: コードの生成
3. **Test（テスト）**: テストの作成と実行

#### Memory Bank（メモリバンク）

Memory Bankは、AI-DLCのファイルベースのストレージシステムです。すべてのプロジェクトアーティファクトを保存し、完全なトレーサビリティを提供します。

**Memory Bankの構造:**
```
memory-bank/
├── standards/           # プロジェクト標準
│   ├── tech-stack.md
│   ├── coding-standards.md
│   ├── system-architecture.md
│   ├── ux-guide.md
│   └── api-conventions.md
├── intents/             # インテントとユニット
│   └── {intent}/
│       ├── intent.md
│       ├── requirements.md
│       ├── context.md
│       └── units/
│           └── {unit}/
│               ├── stories.md
│               └── bolts/
├── bolts/               # ボルト実行記録
└── operations/          # 運用アーティファクト
    ├── environments.md
    ├── runbooks/
    └── monitoring/
```

#### Standards（標準）

Standardsは、プロジェクト全体で一貫性を確保するための基準文書です。

| 標準 | 説明 |
| :--- | :--- |
| **Tech Stack** | 言語、フレームワーク、データベース、インフラの選択 |
| **Coding Standards** | フォーマット、リンティング、命名規則、テスト戦略 |
| **System Architecture** | アーキテクチャスタイル、API設計、状態管理パターン |
| **UX Guide** | デザインシステム、スタイリング、アクセシビリティ |
| **API Conventions** | APIスタイル、バージョニング、レスポンス形式 |

### エージェントシステム

AI-DLC Flowは、各フェーズに特化した4つの専門エージェントによって推進されます。

#### Master Agent

Master Agentは、AI-DLCフローのオーケストレーターです。プロジェクト全体の状態を把握し、適切な専門エージェントにルーティングします。

```
/specsmd-master-agent
```

| コマンド | 説明 |
| :--- | :--- |
| `project-init` | プロジェクトを標準で初期化 |
| `analyze-context` | 現在のプロジェクト状態を表示 |
| `route-request` | 適切なエージェントにルーティング |
| `explain-flow` | AI-DLC方法論を説明 |
| `answer-question` | 質問に回答 |

**使用タイミング:**
- 新しいプロジェクトを開始する時
- どこに行けばいいかわからない時
- 方向性が必要な時
- AI-DLCを学習する時

#### Inception Agent

Inception Agentは、AI-DLCの最初のフェーズを担当します。高レベルの目標を捉え、詳細な要件に展開し、管理可能なUnitに分解します。

```
/specsmd-inception-agent
```

| コマンド | 説明 |
| :--- | :--- |
| `intent-create` | 新しいインテントを作成 |
| `intent-list` | すべてのインテントを一覧表示 |
| `requirements` | インテントの要件を詳細化 |
| `context` | システムコンテキストを定義 |
| `units` | ユニットに分解 |
| `story-create` | ユニットのストーリーを作成 |
| `bolt-plan` | 実装用のBoltを計画 |
| `review` | すべてのInceptionアーティファクトをレビュー |

**ワークフロー:**
```
intent-create → requirements → context → units → story-create → bolt-plan → review
```

#### Construction Agent

Construction Agentは、AI-DLCの2番目のフェーズを担当します。Boltを検証済みのステージを通じて実行し、テスト済みの本番対応コードを生成します。

```
/specsmd-construction-agent
```

| コマンド | 説明 |
| :--- | :--- |
| `bolt-start` | Boltの実行を開始または継続 |
| `bolt-status` | Boltの進捗を確認 |
| `bolt-list` | すべてのBoltを一覧表示 |
| `bolt-replan` | 必要に応じてBoltを再計画 |

#### Operations Agent

Operations Agentは、AI-DLCの3番目のフェーズを担当します。構築された機能を本番環境にデプロイし、正しく動作することを確認し、監視を設定します。

> **注意**: Operations Agentは現在アルファ版です。

```
/specsmd-operations-agent
```

| コマンド | 説明 |
| :--- | :--- |
| `build` | プロジェクトをビルド |
| `deploy` | 環境にデプロイ |
| `verify` | デプロイを検証 |
| `monitor` | 監視を設定 |

### Boltタイプの選択

#### クイック決定ガイド

1. 複雑なビジネスロジックやドメインルールを含む？ → **DDD Construction Bolt**
2. UI、統合、ユーティリティ、または単純なCRUD？ → **Simple Construction Bolt**

#### DDD Construction Boltの使用ケース

- 複雑なルールを持つコアビジネスロジック
- アグリゲート、エンティティ、値オブジェクトを持つドメインモデル
- ビジネス知識をエンコードするサービス
- 誤ったロジックが重大な結果をもたらす機能

**例:**
- Eコマースの注文処理
- 金融計算
- 予約システム
- ワークフローエンジン

#### Simple Construction Boltの使用ケース

- フロントエンドページとコンポーネント
- シンプルなCRUDエンドポイント
- 外部API統合
- ユーティリティとヘルパーモジュール

**例:**
- ユーザープロファイルページ
- REST API統合
- 管理ダッシュボード
- CLIツール

### ベストプラクティス

**Inceptionフェーズ:**
- 明確化質問に詳細に回答する
- 生成されたアーティファクトを必ずレビューする
- Unitが大きすぎると感じたら分割を依頼する
- Constructionに進む前に計画を完了させる

**Constructionフェーズ:**
- Inceptionが完了してから開始する
- チェックポイントで慎重にレビューする
- Boltが長すぎる場合は分割を検討する
- テストステージをスキップしない

**Operationsフェーズ:**
- デプロイ後は必ず検証を行う
- 本番前にステージングでテストする
- 問題が発生する前に監視を設定する
- Runbookを最新の状態に保つ

## IDE拡張機能

### 概要

specsmd IDE拡張機能は、FIRE FlowとAI-DLC Flowの両方の開発進捗を追跡するビジュアルダッシュボードを提供します。IDEのサイドバーからRun/Bolt、仕様、プロジェクトメトリクスを監視できます。

| IDE | サポート状況 |
| :--- | :--- |
| VS Code | 完全サポート |
| Cursor | 完全サポート |
| Google Antigravity | 完全サポート |
| Windsurf | 完全サポート |
| GitHub Codespaces | 完全サポート |
| Gitpod | 完全サポート |
| Amazon Kiro | 完全サポート |

### インストール方法

#### VS Code Marketplace
1. VS Code（またはCursor、Windsurfなど）を開く
2. 拡張機能に移動（`Cmd+Shift+X` / `Ctrl+Shift+X`）
3. "specsmd"を検索
4. **Install**をクリック

または、コマンドラインから:
```bash
code --install-extension fabriqaai.specsmd
```

#### Open VSX Registry
Cursor、Windsurf、Amazon Kiro、Google Antigravity、VSCodium、Gitpod、Google IDX、Eclipse Theia、Code-Server向け。

### 機能

拡張機能は**FIRE FlowとAI-DLC Flowの両方**をサポートします。使用しているフローを自動検出し、インターフェースを適応させます。

#### FIRE Flow Views

`.specs-fire/`フォルダを使用している場合:
- **Runs Tab**: アクティブなRunと完了したRunをファイルリストと共に追跡
- **Intents Tab**: 進捗インジケーター付きでIntentを閲覧
- **Overview Tab**: プロジェクト全体のメトリクスとクイックアクション

#### AI-DLC Flow Views

`memory-bank/`フォルダを使用している場合:

**Bolts Tab**: Bolt実行の進捗をリアルタイムで追跡
- Current Focus: ステージ進捗付きのアクティブなBolt
- Up Next Queue: 開始準備ができているペンディングBolt
- Recent Activity: 完了のタイムライン

**Specs Tab**: プロジェクト仕様を階層的に閲覧
- Intents: 進捗インジケーター付きの高レベル目標
- Units: Intent内の凝集性のある作業要素
- Stories: ステータス付きの個別ユーザーストーリー

**Overview Tab**: 高レベルのプロジェクトメトリクスとクイックアクション
- Progress: 全体の完了率
- Stats: アクティブ、キュー、完了、ブロックされたBolt
- Suggested Actions: AI推奨の次のステップ
- Standards: プロジェクト標準へのクイックアクセス

### アクティベーション

拡張機能は以下を検出すると自動的にアクティベートされます:
- `memory-bank/`フォルダ（AI-DLC Flow）
- `.specs-fire/`フォルダ（FIRE Flow）
- `.specsmd/`フォルダ

### トラブルシューティング

**拡張機能がサイドバーに表示されない**: ワークスペースに必要なフォルダがあることを確認してください。

**データが更新されない**: サイドバーヘッダーの更新アイコンをクリックするか、ファイルを保存してください。

**他のIDEで動作しない**: 正しいレジストリからインストールしてください（VS Code Marketplace vs Open VSX）。

## FAQ

### 一般

**Q: specs.mdとは何ですか？**

A: specs.mdは、AIネイティブな開発フレームワークであり、あらゆるユースケースに対応するプラグイン可能なフローを提供します。Claude Code、Cursor、GitHub Copilotなどの主要なAIコーディングツールと連携します。

**Q: どのフローを選ぶべきですか？**

A: 実行追跡なしで仕様だけが必要ならSimple Flow、複雑さに応じた適応的な厳密性が欲しいならFIRE Flow、包括的なドキュメントと固定ワークフローが必要ならAI-DLC Flowを選択してください。

**Q: specs.mdは無料ですか？**

A: はい、specs.mdはMITライセンスの下でオープンソースです。支払いが必要なのはAI APIトークンのみです。

### Simple Flow

**Q: Simple Flowはいつ使うべきですか？**

A: プロトタイプやMVPの構築、クイックな機能仕様、他のチームへのハンドオフ、実行追跡が不要な場合に使用してください。

### FIRE Flow

**Q: FIRE Flowの実行モードとは何ですか？**

A: FIRE Flowは3つの実行モード（Yolo、Guided、Strict）を提供し、作業の複雑さに応じて監視レベルを調整します。

**Q: FIRE Flowはブラウンフィールドプロジェクトをどのように扱いますか？**

A: FIRE Flowは既存のコードベースを第一級市民として扱います。自動検出、既存モジュールの拡張、既存パターンの尊重、最小限の変更を行います。

**Q: FIRE Flowを使用すべきでない場合は？**

A: 大規模チームでの調整が必要な場合、複雑なドメインロジックでDDDが必要な場合、または規制環境で広範なドキュメントが必要な場合は、AI-DLC Flowを検討してください。

### AI-DLC

**Q: なぜ新しい方法論が必要なのですか？**

A: 従来の方法論は長いイテレーション期間を前提に設計されていました。AIの登場により、イテレーションサイクルは時間や日単位で測定されるようになりました。AIをアジャイルに後付けすることは、その可能性を制限します。

**Q: Intents、Units、Boltsの違いは何ですか？**

A: Intentsは高レベルの目標、Unitsはインテントから派生した凝集性のある作業要素、Boltsは時間制限のある実行セッションです。

**Q: Memory Bankとは何ですか？**

A: Memory Bankは、すべてのプロジェクトアーティファクトを保存するファイルベースのストレージシステムで、完全なトレーサビリティを提供します。

### トラブルシューティング

**Q: エージェントコマンドが認識されません**

A: specs.mdが正しくインストールされていること、および使用しているAIツールに適した起動形式を使用していることを確認してください。

**Q: Memory Bankアーティファクトが見つかりません**

A: 前のBoltアーティファクトが保存されていることを確認してください。`bolt-start`でリロードしてください。

**Q: 生成されたコードが標準に従っていません**

A: `memory-bank/standards/`に標準ファイルが存在し、正しくフォーマットされていることを確認してください。

**Q: エージェントが文脈を忘れてしまう場合はどうすればよいですか？**

A: 各エージェントのセッションは独立しているため、`Memory Bank`や`.specs-fire/state.yaml`に状態が正しく保存されていることを確認してください。エージェントはこれらのファイルから文脈を再読み込みします。

## 付録

### 用語集

| 用語 | 定義 |
| :--- | :--- |
| **Intent** | 最上位の作業単位。ユーザーの高レベルな目標を表す |
| **Unit** | Intentから派生した凝集性のある作業要素 |
| **Story** | Unit内のユーザーストーリー。受け入れ基準を持つ |
| **Bolt** | 時間制限のある集中的な作業セッション |
| **Memory Bank** | すべてのプロジェクトアーティファクトを保存するファイルベースのストレージシステム |
| **Standards** | プロジェクト全体で一貫性を確保するための基準文書 |
| **Work Item** | FIRE Flowにおける作業の基本単位 |
| **Run** | FIRE FlowにおけるWork Itemsの実行セッション |
| **Checkpoint** | 人間の承認が必要な検証ポイント |
| **Walkthrough** | 作業完了後に自動生成されるドキュメント |
| **Constitution** | 常に継承され、決してオーバーライドされない普遍的なポリシー |
| **DDD** | ドメイン駆動設計（Domain-Driven Design） |
| **ADR** | アーキテクチャ決定記録（Architecture Decision Record） |
| **V-Bounce** | 人間が検証者として振る舞うモデル |
| **EARS** | Easy Approach to Requirements Syntax |
