## Gate とは何か

Gate は、**次の工程へ進む前の確認ポイント**です。  
単なる形式的な承認欄ではなく、「この段階で確認すべき品質を満たしているか」を止めて見る仕組みです。

Gate があることで、次のような事故を防ぎやすくなります。

- 目的が曖昧なまま実装に入る
- 受入条件が不足したままテスト設計へ進む
- タスク分解が不十分なまま AI が大量実装を始める
- 承認後の変更が追跡されない

---

## Full Mode の基本

Full Mode では、主に次の Gate を通ります。

| Gate | 主な対象 | 何を確認するか |
|---|---|---|
| Gate 1 | `basic_design.md` | 背景、目的、範囲、制約 |
| Gate 2 | `process.bpmn` | 業務フロー、処理の流れ、関係者の認識合わせ |
| Gate 3 | `spec.md` | ユースケース、受入条件、機能要求 |
| Gate 4 | `plan.md` | テスト戦略、依存関係、リスク、契約 |
| Gate 5 | `tasks.md` | 作業分解、順序、リスクの高い作業の扱い |
| Final | 実装一式 | テスト、証跡、PR 前の準備完了 |

重要なのは、**Gate は前工程の品質を確定するもの**だという点です。  
後工程の都合で急いでも、前工程の曖昧さを残したまま進めると、後でより大きく戻ります。

---

## Lite Mode とは

小規模案件や PoC では、6 段階の Gate が重く感じられることがあります。  
そのため Tecnos-STRIDE には Lite Mode があります。

Lite Mode では、Gate を大きく 3 段階にまとめます。

| Lite Gate | 含まれる内容 |
|---|---|
| Gate A | 基本設計 + フロー |
| Gate B | 仕様 + 計画 |
| Gate C | 実装 + 検証 |

Lite Mode は、次のようなケースで向いています。

- PoC や検証用の小さな機能
- 関係者が少なく、意思決定が早い案件
- 後戻りコストが比較的小さい取り組み

一方で、ERP 連携や監査要件が強い案件では、Full Mode の方が安全です。

---

## Coverage Tier とは

Coverage Tier は、**どれだけの検証と証跡を求めるかの強さ**を示す考え方です。

| Tier | 向いているケース | 期待される検証の重さ |
|---|---|---|
| `critical` | ERP、会計、権限、監査対象 | 最も厳密 |
| `standard` | 通常の業務機能 | 標準的 |
| `starter` | 小規模、試作、限定用途 | 最小限 |

> **注: Coverage Tier**  
> 「テスト件数の多さ」だけを意味しません。  
> 受入条件、契約、証跡、承認の重さまで含めた運用品質の水準です。

---

## なぜ承認は人間だけなのか

Tecnos-STRIDE では、AI は成果物の生成や修正を進められますが、承認は人間だけが行います。  
理由は次の 3 つです。

1. **責任の所在を明確にするため**
2. **業務判断や組織判断を含むため**
3. **AI が自分の成果物を自己承認しないようにするため**

これは AI を信用しないという意味ではありません。  
AI は実行に強く、人間は責任ある判断に強い、という役割分担です。

## Gate を運用するときのコツ

### 1. 承認を「後追いの印鑑」にしない

承認時には、少なくとも「この Gate で見るべき観点」が満たされているかを見ます。  
ファイルが存在するだけで進めると、Gate の意味が薄れます。

### 2. 差分が大きいほど早く止める

不安があるときほど、早い Gate で止める方が安いです。  
Design での 1 時間の見直しは、実装後の 1 週間の手戻りを防ぐことがあります。

### 3. 承認後の変更を軽く見ない

承認後に内容が変わること自体はあります。  
問題は「変わったことが見えない状態」です。必要に応じて再承認や Amendment を使います。

## この章で押さえてほしいこと

- Gate は形式ではなく、品質を止める仕組みです
- Lite Mode は簡略版ですが、責任や承認が不要になるわけではありません
- Coverage Tier はテスト量だけでなく、運用品質の水準も表します
- 承認は人間だけが行い、AI はその前段の実行を支えます
