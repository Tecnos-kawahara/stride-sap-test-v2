# SDD の基本思想

> **対象**: PM / 設計者 / 実行者
> **所要時間**: 15分
> **前提**: [../01_quickstart.md](../01_quickstart.md) を読んでいること
> **In scope**: SDD の狙い、成果物の役割、契約の考え方、トレーサビリティの意味
> **Out of scope**: 各成果物の詳細フィールド、各コマンドの具体的な実行手順

---

## なぜ仕様駆動なのか

従来の開発では、次のような問題が起きやすくなります。

- 要件が会話の中に埋もれてしまい、あとから確認できない
- 実装は進んだが、何を満たせば完了なのかが曖昧になる
- テストや証跡が後回しになり、最終盤で品質不安が噴き出す
- AI を使っても、何を守るべきかが曖昧で暴走しやすい

SDD は、こうした問題に対して**仕様を先に整え、その仕様を正本にする**ことで対処します。

> **注: 仕様駆動**  
> 「先に詳細設計書を大量に書く」という意味ではありません。  
> 何を作るのか、何を満たせばよいか、何を検証するかを、あとから追跡できる形で残す考え方です。

---

## 5 つの主要成果物

Tecnos-STRIDE では、feature ごとに主に 5 つの成果物を扱います。

| 成果物 | 主な役割 | 先に見るべき問い |
|---|---|---|
| `basic_design.md` | 背景、目的、対象範囲、制約を整理する | なぜ作るのか |
| `spec.md` | ユースケースと受入条件を定める | 何ができればよいか |
| `plan.md` | 実装・検証の進め方を決める | どう進めるか |
| `tasks.md` | 実施単位へ分解する | 何から着手するか |
| `evidence_pack.md` | 実施結果と証跡を残す | 何を確認して完了としたか |

これらは独立した文書ではありません。  
ID や参照関係によってつながっており、1 つの feature を別々の角度から説明しています。

---

## トレーサビリティとは何か

トレーサビリティとは、**要件からコード、テスト、証跡までをたどれる状態**です。

たとえば次のようにつながります。

```text
要望
  ↓
basic_design.md の requirement
  ↓
spec.md の use case / acceptance criteria
  ↓
plan.md の test strategy / contract
  ↓
tasks.md の work item
  ↓
実装コード / テスト / evidence_pack
```

このつながりがあると、次の問いに答えやすくなります。

- なぜこの実装が必要なのか
- このテストは何を検証しているのか
- この変更はどの要件に影響するのか
- 承認した内容から何が変わったのか

## 契約（Contract）の考え方

Tecnos-STRIDE でいう契約とは、**システム間や処理間の約束事を、機械が読める形で定義したもの**です。

法的な契約書とは違いますが、「どんな入力を受け、何を返し、どの条件を満たすか」を明文化する点で、約束事という意味合いを持ちます。

よくある契約の例は次の通りです。

| 契約の種類 | 例 |
|---|---|
| API 契約 | OpenAPI で定義された HTTP API |
| イベント契約 | AsyncAPI やイベントスキーマ |
| ファイル契約 | CSV、固定長、EDI、IDoc などの受け渡し形式 |
| DB 契約 | `database_schema.yaml` によるテーブル仕様 |
| CLI 契約 | コマンドの引数、出力、終了コード |

> **注: Contract-First**  
> 先に契約を定めてから実装する考え方です。  
> 先にコードを書いてから「たぶんこう動くはず」と後づけするより、関係者の認識を合わせやすくなります。

## なぜ AI と相性がよいのか

AI は、曖昧な指示よりも、**構造化された仕様**を渡されたときに安定して動きます。

SDD では、AI に対して次のような条件を用意できます。

- 目的と背景が `basic_design.md` にある
- 受入条件が `spec.md` にある
- テスト戦略が `plan.md` にある
- 作業単位が `tasks.md` にある
- 人間の承認ポイントが `APPROVAL.md` にある

つまり、AI は「何を守るべきか」が見えた状態で実行できます。  
これが、単なるチャットベースの依頼よりも再現性が高い理由です。

---

## 成果物は分業のためではなく、認識合わせのためにある

文書が増えると、「結局ドキュメント作業が増えるだけではないか」と感じることがあります。  
しかし Tecnos-STRIDE の成果物は、担当者ごとに別々の書類を作るためのものではありません。

狙いは、**同じ feature を異なる粒度で確認できるようにすること**です。

- PM は目的、リスク、承認条件を見ます
- 設計者はユースケース、契約、業務フローを見ます
- 実装者はタスク、テスト、証跡を見ます

見る角度は違っても、参照している正本は同じです。

## 最低限覚えておきたい用語

| 用語 | 意味 |
|---|---|
| SSoT | 正本。迷ったときに優先して信じる情報源 |
| Gate | 次工程へ進んでよいかを判定する節目 |
| Feature | 実装・検証の基本単位 |
| Epic | 複数 feature を束ねる上位単位 |
| AC | Acceptance Criteria。受入条件 |
| Contract | システム間・処理間の約束事の定義 |
| Evidence | 完了判断の根拠となる証跡 |

---

## この章で押さえてほしいこと

- 仕様は「補足資料」ではなく、開発の中心です
- 契約と受入条件を先に整えると、後工程の手戻りが減ります
- AI を安全に使うには、実行ルールと承認ポイントが必要です
- トレーサビリティがあると、変更の影響を追いやすくなります
