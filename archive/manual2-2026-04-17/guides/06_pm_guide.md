# PM 操作ガイド

> **対象**: PM / PMO / 承認者
> **所要時間**: 15分
> **前提**: [../01_quickstart.md](../01_quickstart.md) と [../concepts/03_phase_gates.md](../concepts/03_phase_gates.md) を読んでいること
> **In scope**: PM が日常運用で見るポイント、承認の進め方、ダッシュボードや GitHub Projects との付き合い方
> **Out of scope**: 各成果物の詳細な執筆方法、実装手順、CI の技術設定

---

## PM の役割をひとことで言うと

Tecnos-STRIDE における PM の主な役割は、**作ることではなく、進めてよいかを判断すること**です。

具体的には次を担います。

- 要件の優先順位を決める
- Gate で承認または差し戻しを判断する
- リスクや依存関係の重さを見極める
- 必要な人に確認を回す
- 進捗が止まっている場所を見つける

---

## PM の 1 日の見方

### 朝: 進捗と詰まりを確認する

最初に見るのは、「誰がどれだけ進んだか」だけではありません。  
むしろ重要なのは、**どこで止まっているか**です。

優先して確認したい項目は次の通りです。

- Gate 承認待ちの feature はないか
- 高リスク feature が `validate` 相当で扱われているか
- 共有契約や依存関係の待ちが発生していないか
- 前日から差し戻しが残っていないか

### 日中: 承認と差し戻しを行う

承認では「細かい書き方」よりも、次の判断軸を重視します。

- 目的と範囲が明確か
- 受入条件が実務上十分か
- テスト戦略がリスクに見合っているか
- 今のまま次工程へ進めて安全か

### 夕方: リスクと学びを確認する

終業前に見るべきなのは、完了数よりも再発防止です。

- 差し戻しが繰り返されている理由は何か
- 教訓として残すべきことはあるか
- 明日詰まりそうな承認待ちはあるか

---

## 承認で見るポイント

### Gate 1, 2

- 背景、対象範囲、制約が誤っていないか
- BPMN や業務フローが現場理解とずれていないか
- 重要な関係者が抜けていないか

### Gate 3, 4

- ユースケースが不足していないか
- 受入条件が曖昧な表現になっていないか
- テスト戦略が案件の重要度に見合っているか
- 依存関係や外部契約が見落とされていないか

### Gate 5, Final

- タスク分解が実行可能な粒度か
- 高リスク作業に十分なレビューがあるか
- テストと証跡が揃っているか
- PR 前チェックで致命的な抜けがないか

---

## 差し戻しをするときの考え方

差し戻しは「全部書き直し」ではありません。  
次に直してほしい観点を、短く具体的に返すのが重要です。

よい差し戻しの例:

- 「背景は理解できますが、対象外範囲が不明確です」
- 「AC はあるが、異常系の受入条件が足りません」
- 「ERP 連携なのに権限・監査観点の記述が薄いです」

避けたい差し戻しの例:

- 「なんとなく不安です」
- 「もう少し丁寧に」
- 「全部見直してください」

## ダッシュボードや可視化の見方

プロジェクトによっては、GitHub Projects やダッシュボード系の補助資料を併用します。  
その場合も、見る順序は変わりません。

1. Gate の停滞
2. 高リスク feature の状態
3. 依存関係の詰まり
4. 承認待ちの偏り

見栄えのよいグラフよりも、**判断が必要な詰まり**を優先してください。

---

## GitHub Projects を使うときの見方

GitHub Projects を使う場合は、Issue や PR の数そのものより、状態の意味づけが重要です。

- `blocked` が増えているなら、承認か依存の問題を疑う
- `running` が長すぎるなら、タスク粒度や Mode の設定を疑う
- 承認待ち Issue が偏るなら、レビュー体制を見直す

> **注: state 管理**  
> Tecnos-STRIDE では、ファイルベースの成果物と、Issue/Projects の運用が並行することがあります。  
> どちらも便利ですが、正本は仕様成果物側にあります。

---

## PM が最低限押さえるコマンド

```bash
# feature の状態確認
sdd-templates/bin/stride phase-status specs/<feature>/

# 次の進め方を確認
sdd-templates/bin/stride auto-continue specs/<feature>/

# feature の整合性確認
sdd-templates/bin/stride lint specs/<feature>/

# PR 前の総合チェック
sdd-templates/bin/stride pr-check .
```

---

## PM 向けチェックリスト

- 今どの Gate で止まっているか把握している
- 差し戻し理由を具体的に伝えられる
- 高リスク案件で Lite Mode を安易に選んでいない
- テストと証跡を見ずに Final 承認していない
- 共有契約や依存の詰まりを見落としていない
